## Testando <a id="testing"></a>

Uma parte importante do processo de desenvolvimento de software é o ato de **testar** o código que escrevemos para garantir que tudo funcione como esperamos. Nesta aula, discutiremos várias maneiras de melhorar a forma como testamos nosso código.

## Assert <a id="assert"></a>

Uma das maneiras mais simples de executar testes em Python é usando o comando `assert`. Este comando é seguido por alguma expressão que deve ser `True`. Se a expressão for `True`, nada acontecerá, e se for `False`, uma exceção será lançada. Vamos ver como poderíamos incorporar o comando para testar a função `square` que escrevemos ao aprender Python. Quando a função é escrita corretamente, nada acontece, pois o `assert` é `True`:

    def square(x):
        return x * x

    assert square(10) == 100

    """ Saída:

    """

E então, quando é escrito incorretamente, uma exceção é lançada.

    def square(x):
        return x + x

    assert square(10) == 100

    """ Saída:
    Traceback (most recent call last):
      Arquivo "assert.py", linha 4, em <módulo>
        assert square(10) == 100
    AssertionError
    """

### Desenvolvimento Orientado a Testes <a id="test-driven-development"></a>

Ao começar a construir projetos maiores, é recomendável considerar o uso do **desenvolvimento orientado a testes**, um estilo de desenvolvimento no qual toda vez que você corrige um bug, você adiciona um teste que verifica esse bug a um conjunto crescente de testes que são executados toda vez que você faz alterações. Isso ajudará a garantir que os recursos adicionais que você adicionar a um projeto não interfiram nos recursos existentes.

Agora, vamos ver uma função um pouco mais complexa e pensar em como escrever testes pode nos ajudar a encontrar erros. Vamos escrever uma função chamada `is_prime` que retorna `True` se e somente se sua entrada for um número primo:

    import math

    def is_prime(n):

        # Sabemos que números menores do que 2 não são primos
        if n < 2:
            return False

        # Verificando os fatores até a raiz quadrada de n
        for i in range(2, int(math.sqrt(n))):

            # Se i for um fator, retorne False
            if n % i == 0:
                return False

        # Se nenhum fator for encontrado, retorne True
        return True

Agora, vamos dar uma olhada em uma função que escrevemos para testar nossa função `prime`:

    from prime import is_prime

    def test_prime(n, expected):
        if is_prime(n) != expected:
            print(f"ERRO em is_prime({n}), esperado {expected}")

Neste ponto, podemos entrar em nosso interpretador Python e testar alguns valores:

    >>> test_prime(5, True)
    >>> test_prime(10, False)
    >>> test_prime(25, False)
    ERRO em is_prime(25), esperado False

Podemos ver a partir da saída acima que 5 e 10 foram corretamente identificados como primos e não primos, mas 25 foi incorretamente identificado como primo, então deve haver algo errado com nossa função. Antes de investigar o que há de errado com nossa função, vamos ver uma maneira de automatizar nossos testes. Uma maneira de fazer isso é criando um **script de shell**, ou algum script que pode ser executado dentro de nosso terminal. Esses arquivos exigem uma extensão `.sh`, então nosso arquivo será chamado de `tests0.sh`. Cada uma das linhas abaixo consiste em

1.  Um `python3` para especificar a versão do Python que estamos executando
2.  Um `-c` para indicar que desejamos executar um comando
3.  Um comando para ser executado no formato de string

```
python3 -c "from tests0 import test_prime; test_prime(1, False)"
python3 -c "from tests0 import test_prime; test_prime(2, True)"
python3 -c "from tests0 import test_prime; test_prime(8, False)"
python3 -c "from tests0 import test_prime; test_prime(11, True)"
python3 -c "from tests0 import test_prime; test_prime(25, False)"
python3 -c "from tests0 import test_prime; test_prime(28, False)"
```

Agora podemos executar esses comandos rodando `./tests0.sh` em nosso terminal, nos dando este resultado:

    ERRO em is_prime(8), esperado False
    ERRO em is_prime(25), esperado False

## Teste de Unidade <a id="unit-testing"></a>

Apesar de termos conseguido rodar os testes automaticamente usando o método acima, podemos ainda querer evitar ter que escrever cada um desses testes. Felizmente, podemos usar a biblioteca Python `unittest` para tornar esse processo um pouco mais fácil. Vamos ver como um programa de teste pode ser para nossa função `is_prime`.

```python
# Importar a biblioteca unittest e nossa função
import unittest
from prime import is_prime

# Uma classe contendo todos os nossos testes
class Tests(unittest.TestCase):

    def test_1(self):
        """Verificar que 1 não é primo."""
        self.assertFalse(is_prime(1))

    def test_2(self):
        """Verificar que 2 é primo."""
        self.assertTrue(is_prime(2))

    def test_8(self):
        """Verificar que 8 não é primo."""
        self.assertFalse(is_prime(8))

    def test_11(self):
        """Verificar que 11 é primo."""
        self.assertTrue(is_prime(11))

    def test_25(self):
        """Verificar que 25 não é primo."""
        self.assertFalse(is_prime(25))

    def test_28(self):
        """Verificar que 28 não é primo."""
        self.assertFalse(is_prime(28))


# Rodar cada uma das funções de teste
if __name__ == "__main__":
    unittest.main()
```

Observe que cada uma das funções dentro da nossa classe `Tests` segue um padrão:

- O nome das funções começa com `test_`. Isso é necessário para as funções serem executadas automaticamente com a chamada para `unittest.main()`.
- Cada teste recebe o argumento `self`. Isso é padrão ao escrever métodos dentro de classes Python.
- A primeira linha de cada função contém uma **string de documentação** cercada por três aspas. Isso não é apenas para a legibilidade do código. Quando os testes são executados, o comentário será exibido como uma descrição do teste se ele falhar.
- A próxima linha de cada uma das funções contém uma assertiva no formato `self.assertALGO`. Existem muitas assertivas diferentes que você pode utilizar, incluindo `assertTrue`, `assertFalse`, `assertEqual` e `assertGreater`. Você pode encontrar essas e outras checando a [documentação](https://docs.python.org/3/library/unittest.html#unittest.TestCase.assertEqual).

Agora, vamos conferir os resultados destes testes:

```python
...F.F
======================================================================

Verificar que 25 não é primo.
----------------------------------------------------------------------

    File "tests1.py", line 26, in test_25
    self.assertFalse(is_prime(25))

======================================================================
FAIL: test_8 (__main__.Tests)
Verificar que 8 não é primo.

```

Após a execução dos testes, o `unittest` nos fornece informações úteis sobre o que foi encontrado. Na primeira linha, ele nos dá uma série de `.` para sucessos e `F`s para falhas na ordem em que os testes foram escritos.

    ...F.F

A seguir, para cada teste que falhou, nós recebemos o nome da função que falhou:

    FAIL: test_25 (__main__.Tests)

o comentário descritivo que fornecemos anteriormente:

    Verificar que 25 não é primo.

E uma rastreabilidade para a exceção:

    Traceback (most recent call last):
      File "tests1.py", line 26, in test_25
        self.assertFalse(is_prime(25))
    AssertionError: True não é falso

E por fim, nos é mostrado um resumo de quantos testes foram executados, quanto tempo levaram e quantos falharam:

    Rodou 6 testes em 0.001s

    FALHO (falhas=2)

Agora vamos corrigir o erro em nossa função. Descobrimos que precisamos testar um número adicional em nosso laço `for`. Por exemplo, quando `n` é `25`, a raiz quadrada é `5`, mas quando este é um argumento na função `range`, o laço `for` termina no número `4`. Portanto, podemos simplesmente alterar o cabeçalho de nosso laço `for` para:

    for i in range(2, int(math.sqrt(n)) + 1):

Agora, quando rodamos os testes novamente utilizando nossos testes de unidade, obtemos a seguinte saída, indicando que nossa alteração corrigiu o bug.

    ......
    ----------------------------------------------------------------------
    Rodou 6 testes em 0.000s

    OK

Estes testes automatizados se tornarão ainda mais úteis conforme você trabalha para otimizar esta função. Por exemplo, você pode querer usar o fato de que não é necessário verificar todos os inteiros como fatores, somente primos menores (se um número não é divisível por 3, ele também não é divisível por 6, 9, 12, ...), ou você pode querer usar testes de primalidade mais avançados, como os testes de primalidade [Fermat](https://en.wikipedia.org/wiki/Fermat_primality_test) e [Miller-Rabin](https://en.wikipedia.org/wiki/Miller%E2%80%93Rabin_primality_test). Sempre que fizer alterações para melhorar esta função, você vai querer a capacidade de rodar facilmente seus testes de unidade novamente para garantir que sua função ainda está correta.

## Testando o Django

Agora, vamos ver como podemos aplicar as ideias de teste automatizado ao criar aplicações Django. Ao trabalhar com isso, estaremos usando o projeto `flights` que criamos quando aprendemos pela primeira vez sobre os modelos do Django. Primeiro, vamos adicionar um método ao nosso modelo `Flight` que verifica se um voo é válido verificando duas condições:

1. A origem não é a mesma que o destino
2. A duração é maior que 0 minutos

Agora, nosso modelo pode se parecer com isto:

```python
class Flight(models.Model):
    origin = models.ForeignKey(Airport, on_delete=models.CASCADE, related_name="departures")
    destination = models.ForeignKey(Airport, on_delete=models.CASCADE, related_name="arrivals")
    duration = models.IntegerField()

    def __str__(self):
        return f"{self.id}: {self.origin} para {self.destination}"

    def is_valid_flight(self):
        return self.origin != self.destination or self.duration > 0
```

Para garantir que nossa aplicação funcione conforme o esperado, sempre que criamos uma nova aplicação, automaticamente recebemos um arquivo `tests.py`. Ao abrir este arquivo pela primeira vez, veremos que a biblioteca [TestCase](https://docs.djangoproject.com/en/4.0/topics/testing/overview/) do Django é importada automaticamente:

```python
from django.test import TestCase
```

Uma vantagem de usar a biblioteca `TestCase` é que ao executarmos nossos testes, um novo banco de dados será criado para fins de teste apenas. Isso é útil, pois evitamos o risco de modificar ou excluir acidentalmente entradas existentes em nosso banco de dados e não precisamos nos preocupar com a remoção de entradas falsas que criamos apenas para testes.

Para começar a usar esta biblioteca, primeiro queremos importar todos os nossos modelos:

```python
from .models import Flight, Airport, Passenger
```

E então, criaremos uma nova classe que estende a classe `TestCase` que acabamos de importar. Dentro desta classe, definiremos uma função `setUp` que será executada no início do processo de teste. Nesta função, provavelmente iremos criar. Veja como nossa classe será para começar:

```python
class FlightTestCase(TestCase):

    def setUp(self):

        a1 = Airport.objects.create(code="AAA", city="Cidade A")
        a2 = Airport.objects.create(code="BBB", city="Cidade B")

        # Criar voos.
        Flight.objects.create(origin=a1, destination=a2, duration=100)
        Flight.objects.create(origin=a1, destination=a1, duration=200)
        Flight.objects.create(origin=a1, destination=a2, duration=-100)
```

Agora que temos algumas entradas em nosso banco de dados de teste, adicionaremos algumas funções a esta classe para realizar alguns testes. Primeiro, vamos garantir que nossos campos `departures` e `arrivals` funcionem corretamente tentando contar o número de partidas (que sabemos que devem ser 3) e chegadas (que devem ser 1) do aeroporto `AAA`:

```python
def test_departures_count(self):

def test_arrivals_count(self):
```

Também podemos testar a função `is_valid_flight` que adicionamos ao nosso modelo `Flight`. Começaremos afirmando que a função retorna verdadeiro quando o voo é válido:

```python
def test_valid_flight(self):
    a1 = Airport.objects.get(code="AAA")
```

### Testando o Cliente

Ao criar aplicações web, provavelmente desejaremos verificar não apenas se funções específicas funcionam, mas também se as páginas da web individuais são carregadas conforme o esperado. Podemos fazer isso criando um objeto `Client` em nossa classe de teste do Django e depois fazendo solicitações usando esse objeto. Para fazer isso, primeiro precisamos adicionar `Client` aos nossos imports:

```python
from django.test import Client, TestCase
```

Por exemplo, vamos adicionar um teste que garanta que obtemos um código de resposta HTTP de 200 e que todos os três de nossos voos sejam adicionados ao contexto de uma resposta:

```python
def test_index(self):

    # Configurar cliente para fazer solicitações
    c = Client()

    # Enviar solicitação get para a página de índice e armazenar a resposta
    response = c.get("/flights/")

    # Garantir que o código de status seja 200
    self.assertEqual(response.status_code, 200)

    # Garantir que três voos sejam retornados no contexto
    self.assertEqual(response.context["flights"].count(), 3)
```

Podemos verificar de forma semelhante se obtemos um código de resposta válido para uma página de voo válida e um código de resposta inválido para uma página de voo que não existe. (Observe que usamos a função `Max` para encontrar o `id` máximo, ao incluir `from django.db.models import Max` no topo do nosso arquivo)

```python
def test_valid_flight_page(self):
    a1 = Airport.objects.get(code="AAA")
    f = Flight.objects.get(origin=a1, destination=a1)

    c = Client()
    response = c.get(f"/flights/{f.id}")
    self.assertEqual(response.status_code, 200)

def test_invalid_flight_page(self):
    max_id = Flight.objects.all().aggregate(Max("id"))["id__max"]

    c = Client()
    response = c.get(f"/flights/{max_id + 1}")
    self.assertEqual(response.status_code, 404)
```

Finalmente, vamos adicionar alguns testes para garantir que as listas de passageiros e não passageiros estejam sendo geradas conforme o esperado:

```python
def test_flight_page_passengers(self):
    f = Flight.objects.get(pk=1)
    p = Passenger.objects.create(first="Alice", last="Adams")
    f.passengers.add(p)

    c = Client()
    response = c.get(f"/flights/{f.id}")
    self.assertEqual(response.status_code, 200)
    self.assertEqual(response.context["passengers"].count(), 1)

def test_flight_page_non_passengers(self):
    f = Flight.objects.get(pk=1)
    p = Passenger.objects.create(first="Alice", last="Adams")

    c = Client()
    response = c.get(f"/flights/{f.id}")
    self.assertEqual(response.status_code, 200)
    self.assertEqual(response.context["non_passengers"].count(), 1)
```

## Selenium <a id="selenium"></a>

Até agora, fomos capazes de testar o código do lado do servidor que escrevemos usando Python e Django, mas conforme vamos desenvolvendo nossas aplicações, vamos querer a capacidade de criar testes para o código do lado do cliente também. Por exemplo, vamos voltar à nossa página `counter.html` e trabalhar na escrita de alguns testes para ela.

Vamos começar escrevendo uma página de contador ligeiramente diferente, onde incluímos um botão para diminuir a contagem:

    <!DOCTYPE html>
    <html lang="en">
        <head>
            <title>Contador</title>
            <script>

                // Aguardar o carregamento da página
                document.addEventListener('DOMContentLoaded', () => {

                    // Inicializar a variável como 0
                    let contador = 0;

                    // Se o botão de incremento for clicado, incrementar o contador e alterar o html interno
                    document.querySelector('#increase').onclick = () => {
                        contador ++;
                        document.querySelector('h1').innerHTML = contador;
                    }

                    // Se o botão de decremento for clicado, decrementar o contador e alterar o html interno
                    document.querySelector('#decrease').onclick = () => {
                        contador --;
                        document.querySelector('h1').innerHTML = contador;
                    }
                })
            </script>
        </head>
        <body>
            <h1>0</h1>
            <button id="increase">+</button>
            <button id="decrease">-</button>
        </body>
    </html>

Agora, se desejarmos testar este código, poderíamos simplesmente abrir nosso navegador da web, clicar nos dois botões e observar o que acontece. No entanto, isso se tornaria muito tedioso conforme escrevemos aplicações únicas maiores e maiores, é por isso que vários frameworks foram criados para ajudar nos testes no navegador, sendo um deles o [Selenium](https://www.selenium.dev/).

Usando o Selenium, poderemos definir um arquivo de teste em Python onde podemos simular um usuário abrindo um navegador da web, navegando até nossa página e interagindo com ela. Nossa principal ferramenta ao fazer isso é conhecida como **Driver Web**, que abrirá um navegador da web em seu computador. Vejamos como poderíamos começar a usar essa biblioteca para interagir com páginas. Note que abaixo usamos tanto `selenium` quanto `ChromeDriver`. O Selenium pode ser instalado para Python executando `pip install selenium` e o `ChromeDriver` pode ser instalado executando `pip install chromedriver-py`.

    import os
    import pathlib
    import unittest

    from selenium import webdriver

    # Encontra o Identificador de Recurso Uniforme de um arquivo
    def file_uri(filename):
        return pathlib.Path(os.path.abspath(filename)).as_uri()

    # Configura o driver da web usando o Google Chrome
    driver = webdriver.Chrome()

O código acima é toda a configuração básica de que precisamos, então agora podemos ir para usos mais interessantes empregando o interpretador Python. Uma observação sobre as primeiras linhas é que, para direcionar uma página específica, precisamos do **Identificador de Recurso Uniforme (URI)** dessa página, que é uma cadeia única que representa esse recurso.

    # Encontra o URI do nosso arquivo recém-criado
    >>> uri = file_uri("counter.html")

    # Usa o URI para abrir a página da web
    >>> driver.get(uri)

    # Acessa o título da página atual
    >>> driver.title
    'Contador'

    # Acessa o código-fonte da página
    >>> driver.page_source
    '<html lang="en"><head>\n        <title>Contador</title>\n        <script>\n            \n            // Aguardar o carregamento da página\n            document.addEventListener(\'DOMContentLoaded\', () => {\n\n                // Inicializar a variável como 0\n                let contador = 0;\n\n                // Se o botão de incremento for clicado, incrementar o contador e alterar o html interno\n                document.querySelector(\'#increase\').onclick = () => {\n                    contador ++;\n                    document.querySelector(\'h1\').innerHTML = contador;\n                }\n\n                // Se o botão de decremento for clicado, decrementar o contador e alterar o html interno\n                document.querySelector(\'#decrease\').onclick = () => {\n                    contador --;\n                    document.querySelector(\'h1\').innerHTML = contador;\n                }\n            })\n        </script>\n    </head>\n    <body>\n        <h1>0</h1>\n        <button id="increase">+</button>\n        <button id="decrease">-</button>\n    \n</body></html>'

    # Encontra e armazena os botões de aumento e diminuição:
    >>> aumento = driver.find_element_by_id("increase")
    >>> diminuicao = driver.find_element_by_id("decrease")

    # Simula o usuário clicando nos dois botões
    >>> aumento.click()
    >>> aumento.click()
    >>> diminuicao.click()

    # Podemos até incluir cliques em outras construções Python:
    >>> for i in range(25):
    ...     aumento.click()

Agora vamos dar uma olhada em como podemos usar essa simulação para criar testes automatizados da nossa página:

    # Esquema padrão da classe de teste
    class TestesDaPaginaWeb(unittest.TestCase):

        def test_titulo(self):
            """Certificar-se de que o título está correto"""
            driver.get(file_uri("counter.html"))
            self.assertEqual(driver.title, "Contador")

        def test_aumento(self):
            """Certificar-se de que o cabeçalho é atualizado para 1 após 1 clique no botão de aumento"""
            driver.get(file_uri("counter.html"))
            aumento = driver.find_element_by_id("increase")
            aumento.click()
            self.assertEqual(driver.find_element_by_tag_name("h1").text, "1")

        def test_diminuicao(self):
            """Certificar-se de que o cabeçalho é atualizado para -1 após 1 clique no botão de diminuição"""
            driver.get(file_uri("counter.html"))
            diminuicao = driver.find_element_by_id("decrease")
            diminuicao.click()
            self.assertEqual(driver.find_element_by_tag_name("h1").text, "-1")

        def test_multiplo_aumento(self):
            """Certificar-se de que o cabeçalho é atualizado para 3 após 3 cliques no botão de aumento"""
            driver.get(file_uri("counter.html"))
            aumento = driver.find_element_by_id("increase")
            for i in range(3):
                aumento.click()
            self.assertEqual(driver.find_element_by_tag_name("h1").text, "3")

    if __name__ == "__main__":
        unittest.main()

Agora, se executarmos `python tests.py`, nossas simulações serão realizadas no navegador e os resultados dos testes serão impressos no console. Eis um exemplo de como isso pode parecer quando há um bug no código e um teste falha:

## CI/CD <a id="cicd"></a>

**CI/CD**, que significa **Integração Contínua e Entrega Contínua**, é um conjunto de práticas recomendadas de desenvolvimento de software que ditam como o código é escrito por uma equipe e como esse código é posteriormente entregue aos usuários da aplicação. Como o nome sugere, esse método consiste em duas partes principais:

- Integração Contínua:
  - Fusões frequentes com o ramo principal
  - Testes unitários automatizados a cada fusão
- Entrega Contínua:
  - Cronogramas de lançamento curtos, significando que novas versões de uma aplicação são lançadas com frequência.

O CI/CD tornou-se cada vez mais popular entre equipes de desenvolvimento de software por várias razões:

- Quando diferentes membros da equipe estão trabalhando em diferentes funcionalidades, muitos problemas de compatibilidade podem surgir quando várias funcionalidades são combinadas ao mesmo tempo. A integração contínua permite que as equipes resolvam pequenos conflitos à medida que surgem.
- Porque os testes unitários são executados a cada fusão, quando um teste falha, é mais fácil isolar a parte do código que está causando o problema.
- O lançamento frequente de novas versões de uma aplicação permite que os desenvolvedores isolem problemas se surgirem após o lançamento.
- O lançamento de mudanças pequenas e incrementais permite que os usuários se acostumem lentamente às novas funcionalidades do aplicativo em vez de serem sobrecarregados com uma versão totalmente diferente.
- Não esperar para lançar novas funcionalidades permite que as empresas se mantenham à frente em um mercado competitivo.

## GitHub Actions <a id="github-actions"></a>

Uma ferramenta popular usada para auxiliar na integração contínua é conhecida como [GitHub Actions](https://github.com/features/actions). O GitHub Actions nos permitirá criar fluxos de trabalho onde podemos especificar determinadas ações a serem executadas toda vez que alguém faz um push para um repositório git. Por exemplo, podemos querer verificar a cada push se um guia de estilo está sendo seguido, ou se um conjunto de testes unitários é aprovado.

Para configurar uma ação do GitHub, usaremos uma linguagem de configuração chamada **YAML**. O YAML estrutura seus dados em torno de pares chave-valor (como um objeto JSON ou um Dicionário Python). Aqui está um exemplo de um arquivo YAML simples:

    chave1: valor1
    chave2: valor2
    chave3:
        - item1
        - item2
        - item3

Agora, vamos ver um exemplo de como configurar um arquivo YAML (que tem a forma `nome.yml` ou `nome.yaml`) que funcione com o GitHub Actions. Para fazer isso, criarei um diretório `.github` no meu repositório, e em seguida um diretório `workflows` dentro dele, e finalmente um arquivo `ci.yml` dentro deste último. Nesse arquivo, escreveremos:

    nome: Testando
    em: push

    jobs:
      test_project:
        runs-on: ubuntu-último
        steps:
        - uses: actions/checkout@v2
        - name: Executar testes unitários Django
          run: |
            pip3 install --user django
            python3 manage.py test

Como é nossa primeira vez usando o GitHub Actions, vamos explicar o que cada parte deste arquivo está fazendo:

- Primeiro, damos ao fluxo de trabalho um `nome`, que no nosso caso é "Testando".
- Em seguida, com a chave `em`, especificamos quando o fluxo de trabalho deve ser executado. No nosso caso, desejamos realizar os testes sempre que alguém fizer um push para o repositório.
- O restante do arquivo está contido em uma chave `jobs`, que indica quais trabalhos devem ser executados a cada push.
  - No nosso caso, o único trabalho é `test_project`. Cada trabalho deve definir duas partes:
    - A chave `runs-on` especifica em qual das máquinas virtuais do GitHub gostaríamos que nosso código fosse executado.
    - A chave `steps` fornece as ações que devem ocorrer quando este trabalho é executado.
      - Na chave `uses`, especificamos qual ação do GitHub queremos usar. `actions/checkout@v2` é uma ação escrita pelo GitHub que podemos utilizar.
      - A chave `name` permite fornecer uma descrição da ação que estamos realizando.
      - Depois da chave `run`, digitamos os comandos que desejamos executar no servidor do GitHub. No nosso caso, desejamos instalar o Django e então executar o arquivo de teste.

Agora, vamos abrir nosso repositório no GitHub e dar uma olhada em algumas das abas perto do topo da página:

- **Código**: Esta é a aba que temos usado com mais frequência, pois nos permite visualizar os arquivos e pastas dentro do nosso diretório.
- **Issues**: Aqui podemos abrir e fechar problemas, que são solicitações para correções de bugs ou novas funcionalidades. Podemos pensar nisso como uma lista de tarefas para nossa aplicação.
- **Pull Requests**: Solicitações de pessoas que desejam mesclar algum código de um branch em outro. Esta é uma ferramenta útil, pois permite que as pessoas realizem **revisões de código** onde comentam e fornecem sugestões antes que o código seja integrado ao branch principal.
- **GitHub Actions**: Esta é a aba que usaremos ao trabalhar com integração contínua, pois fornece logs das ações que foram realizadas após cada push.

Aqui, vamos imaginar que fizemos um push em nossas alterações _antes_ de corrigir o bug que tínhamos na função `is_valid_flight` em `models.py` dentro do nosso projeto `airport`. Agora podemos navegar até a aba **GitHub Actions**, clicar no nosso push mais recente, clicar na ação que falhou e visualizar o log:

## Docker <a id="docker"></a>

Problemas podem surgir no mundo do desenvolvimento de software quando a configuração em seu computador é diferente daquela em que sua aplicação está sendo executada. Você pode ter uma versão diferente do Python ou alguns pacotes adicionais instalados que permitem que a aplicação rode sem problemas em seu computador, enquanto ela poderia travar no servidor. Para evitar esses problemas, precisamos de uma maneira de garantir que todos que trabalham em um projeto estejam usando o mesmo ambiente. Uma maneira de fazer isso é usar uma ferramenta chamada **Docker**, que é um software de containerização, significando que ele cria um ambiente isolado em seu computador que pode ser padronizado entre muitos colaboradores e o servidor no qual seu site é executado. Embora o Docker seja um pouco como uma **Máquina Virtual**, na verdade são tecnologias diferentes. Uma máquina virtual (como a usada no GitHub Actions ou ao lançar um servidor da [AWS](./) é efetivamente um computador virtual completo com seu próprio sistema operacional, o que acaba ocupando muito espaço onde quer que esteja sendo executada. Os Dockers, por outro lado, funcionam configurando um contêiner dentro de um computador existente, ocupando assim menos espaço.

Agora que temos uma ideia do que é um contêiner Docker, vamos ver como podemos configurar um em nossos computadores. Nosso primeiro passo para fazer isso será criar um **Arquivo Docker** que chamaremos de `Dockerfile`. Dentro deste arquivo, vamos fornecer instruções sobre como criar uma **Imagem Docker** que descreve as bibliotecas e binários que desejamos incluir em nosso contêiner. Aqui está um exemplo de como nosso `Dockerfile` pode ser:

    FROM python:3
    COPY .  /usr/src/app
    WORKDIR /usr/src/app
    RUN pip install -r requirements.txt
    CMD ["python3", "manage.py", "runserver", "0.0.0.0:8000"]

Aqui, vamos dar uma olhada detalhada em o que o arquivo acima realmente faz:

- `FROM python3`: isso mostra que estamos baseando esta imagem em uma imagem padrão na qual o Python 3 está instalado. Isso é bastante comum ao escrever um Arquivo Docker, pois permite evitar o trabalho de redefinir o mesmo setup básico com cada nova imagem.
- `COPY . /usr/src/app`: Isso mostra que queremos copiar tudo do nosso diretório atual (`.`) e armazená-lo no diretório `/usr/src/app` em nosso novo contêiner.
- `WORKDIR /usr/src/app`: Isso define onde executaremos comandos dentro do contêiner. (Algo como `cd` no terminal)
- `RUN pip install -r requirements.txt`: Nesta linha, assumindo que você incluiu todos os seus requisitos em um arquivo chamado `requirements.txt`, todos eles serão instalados dentro do contêiner.
- `CMD ["python3", "manage.py", "runserver", "0.0.0.0:8000"]`: Por fim, especificamos o comando que deve ser executado ao iniciar o contêiner.

Até agora, nesta aula, só temos usado SQLite, pois é o sistema de gerenciamento de banco de dados padrão para o Django. Contudo, em aplicações ao vivo com usuários reais, o SQLite é quase nunca usado, pois não é tão facilmente escalável quanto outros sistemas. Felizmente, se desejamos executar um servidor separado para nosso banco de dados, podemos simplesmente adicionar outro contêiner Docker e executá-los juntos usando um recurso chamado **Docker Compose**. Isso permitirá que dois servidores diferentes sejam executados em contêineres separados, mas também possam se comunicar entre si. Para especificar isso, usaremos um arquivo YAML chamado `docker-compose.yml`:

    version: '3'

    services:
        db:
            image: postgres

        web:
            build: .
            volumes:
                - .:/usr/src/app
            ports:
                - "8000:8000"

No arquivo acima:

- Especificamos que estamos usando a versão 3 do Docker Compose
- Destacamos dois serviços:
  - `db` configura nosso contêiner de banco de dados com base em uma imagem já escrita pelo Postgres.
  - `web` configura o contêiner de nosso servidor instruindo o Docker a:
    - Usar o Dockerfile dentro do diretório atual.
    - Usar o caminho especificado dentro do contêiner.
    - Vincular a porta 8000 dentro do contêiner à porta 8000 em nosso computador.

Agora estamos prontos para iniciar nossos serviços com o comando `docker-compose up`. Isso iniciará ambos os servidores em novos contêineres Docker.

Neste ponto, talvez queiramos executar comandos dentro de nosso contêiner Docker para adicionar entradas ao banco de dados ou executar testes. Para fazer isso, primeiro executaremos `docker ps` para mostrar todos os contêineres Docker em execução. Em seguida, encontraremos o `CONTAINER ID` do contêiner que desejamos acessar e executaremos `docker exec -it CONTAINER_ID bash -l`. Isso o levará para o diretório `usr/src/app` que configuramos em nosso contêiner. Podemos executar quaisquer comandos desejados dentro desse contêiner e, em seguida, sair executando `CTRL-D`.
