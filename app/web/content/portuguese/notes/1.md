## Introdução <a id="introduction"></a>

Bem-vindo de volta à aula 1! Na aula 0, introduzimos o HTML, CSS e Sass como ferramentas que podemos usar para criar algumas páginas da web básicas. Hoje, vamos aprender sobre como utilizar o Git e o GitHub para nos ajudar no desenvolvimento de aplicações de programação web.

## Git <a id="git"></a>

- [Git](https://git-scm.com/) é uma ferramenta de linha de comando que nos ajudará com o controle de versão de várias maneiras diferentes:

  - Nos permite acompanhar as mudanças que fazemos em nosso código, salvando instantâneos do nosso código em um determinado momento.

  ![Arquivo em Mudança](https://cs50.harvard.edu/web/2020/notes/1/images/change_file.png)

  - Nos permite sincronizar facilmente o código entre diferentes pessoas que trabalham no mesmo projeto, permitindo que várias pessoas extraiam informações e enviem informações para um repositório armazenado na web.

  ![Múltiplos Usuários](https://cs50.harvard.edu/web/2020/notes/1/images/mult_users.png)

  - Nos permite fazer alterações e testar o código em um _branch_ diferente sem alterar nossa base de código principal e depois mesclar os dois juntos.
  - Nos permite reverter para versões anteriores do nosso código se percebermos que cometemos um erro.

- Nas explicações acima, usamos a palavra **repositório**, que ainda não explicamos. Um repositório Git é um local de arquivo onde armazenaremos todos os arquivos relacionados a um determinado projeto. Estes podem ser remotos (armazenados online) ou locais (armazenados em seu computador).

## GitHub <a id="github"></a>

- O [GitHub](https://www.github.com) é um site que nos permite armazenar repositórios Git remotamente na web.
- Vamos começar criando um novo repositório online

  1.  Certifique-se de ter uma conta no GitHub configurada. Se você ainda não tiver uma, pode criar uma [aqui](https://github.com/join?ref_cta=Sign+up&ref_loc=header+logged+out&ref_page=%2F&source=header-home).
  2.  Clique no **+** no canto superior direito e depois em “New repository”
  3.  Crie um nome de repositório que descreva seu projeto
  4.  (Opcional) Forneça uma descrição para seu repositório
  5.  Escolha se o repositório deve ser público (visível para todos na web) ou privado (visível apenas para você e outras pessoas às quais você conceder acesso)
  6.  (Opcional) Decida se deseja adicionar um README, que é um arquivo descrevendo seu novo repositório.

  ![Novo repositório demo](https://cs50.harvard.edu/web/2020/notes/1/images/new_repo.png)

- Quando tivermos um repositório, provavelmente desejaremos adicionar alguns arquivos a ele. Para fazer isso, pegaremos nosso repositório _remoto_ recém-criado e criaremos uma cópia, ou clone, dele como um repositório _local_ em nosso computador.

  1.  Certifique-se de ter o git instalado em seu computador digitando `git` no seu terminal. Se não estiver instalado, você pode baixá-lo [aqui](https://git-scm.com/downloads).
  2.  Clique no botão verde “Clone or Download” na página do seu repositório e copie a URL que aparece. Se você não criou um README, este link aparecerá perto do topo da página na seção “Quick Setup”.

      ![clonar e adicionar](https://cs50.harvard.edu/web/2020/notes/1/images/clone_and_add.png)

  3.  No seu terminal, execute `git clone <url do repositório>`. Isso irá baixar o repositório para o seu computador. Se você não criou um README, você receberá o aviso: `You appear to have cloned into an empty repository.` Isso é normal e não há com o que se preocupar.

      ![demo de clonagem](https://cs50.harvard.edu/web/2020/notes/1/images/clone.png)

  4.  Execute `ls`, que é um comando que lista todos os arquivos e pastas no seu diretório atual. Você deverá ver o nome do repositório que acabou de clonar.
  5.  Execute `cd <nome do repositório>` para mudar para o diretório dessa pasta.
  6.  Execute `touch <novo nome de arquivo>` para criar um novo arquivo nessa pasta. Agora você pode fazer edições nesse arquivo. Alternativamente, você pode abrir a pasta em seu editor de texto e adicionar novos arquivos manualmente.
  7.  Agora, para informar ao Git que ele deve estar monitorando o novo arquivo que você criou, execute `git add <novo nome de arquivo>` para monitorar esse arquivo específico, ou `git add .` para monitorar todos os arquivos dentro desse diretório.


      ![-am no mesmo tempo](https://cs50.harvard.edu/web/2020/notes/1/images/create_and_add.png)

## Commits <a id="commits"></a>

- Agora, vamos começar a explorar para o que o Git pode ser realmente útil. Depois de fazer algumas alterações em um arquivo, podemos _commitar_ essas mudanças, fazendo uma captura do estado atual do nosso código. Para fazer isso, executamos: `git commit -m "alguma mensagem"` onde a mensagem descreve as alterações que acabamos de fazer.
- Após essa alteração, podemos executar `git status` para ver como nosso código se compara ao código no repositório remoto.
- Quando estivermos prontos para publicar nossos commits locais no Github, podemos executar `git push`. Agora, ao acessarmos o GitHub em nosso navegador, nossas alterações serão refletidas.
- Se você apenas alterou arquivos existentes e não criou novos, em vez de usar `git add .` e depois `git commit...`, podemos condensar isso em um único comando: `git commit -am "alguma mensagem"`. Este comando irá committar todas as alterações que você fez.
- Às vezes, o repositório remoto no GitHub estará mais atualizado do que a versão local. Nesse caso, você deseja primeiro commitar quaisquer alterações e em seguida executar `git pull` para puxar quaisquer alterações remotas para o seu repositório.

## Conflitos de Mesclagem <a id="merge-conflicts"></a>

- Um problema que pode surgir ao trabalhar com o Git, especialmente quando se está colaborando com outras pessoas, é algo chamado **conflito de mesclagem**. Um conflito de mesclagem ocorre quando duas pessoas tentam alterar um arquivo de maneiras que entram em conflito uma com a outra.
- Isso normalmente ocorre quando você faz `git push` ou `git pull`. Quando isso acontece, o Git automaticamente altera o arquivo para um formato que claramente delineia qual é o conflito. Aqui está um exemplo em que a mesma linha foi adicionada de duas maneiras diferentes:

```
a = 1
<<<<<<< HEAD
b = 2
=======
b = 3

>>>>>>> 56782736387980937883
c = 3
d = 4
e = 5
```

- No exemplo acima, você adicionou a linha `b = 2` e outra pessoa escreveu `b = 3`, agora precisamos escolher qual manter. O número longo é um _hash_ que representa o commit que está entrando em conflito com suas edições. Muitos editores de texto também oferecem destaque e opções simples como "aceitar atual" ou "aceitar entrante" que economizam o tempo de deletar as linhas adicionadas acima.
- Outro comando Git potencialmente útil é `git log`, que fornece um histórico de todos os seus commits naquele repositório.

![Git Log](https://cs50.harvard.edu/web/2020/notes/1/images/git_log.png)

- Potencialmente ainda mais útil, se você perceber que cometeu um erro, você pode reverter para um commit anterior usando o comando `git reset` de duas maneiras:
  - `git reset --hard <commit>` reverte seu código exatamente como estava após o commit especificado. Para especificar o commit, use o hash do commit associado, que pode ser encontrado usando o `git log` conforme mostrado acima.
  - `git reset --hard origin/master` reverte seu código para a versão atualmente armazenada online no Github.

## Ramificação <a id="branching"></a>

Depois de trabalhar em um projeto por um tempo, você pode decidir que deseja adicionar um recurso adicional. No momento, talvez apenas faça commit das alterações para este novo recurso conforme mostrado na imagem abaixo

![Sem Ramificação](https://cs50.harvard.edu/web/2020/notes/1/images/no_branch.png)

Mas isso pode se tornar problemático se descobrirmos um bug em nosso código original e quisermos reverter sem alterar o novo recurso. Aqui é onde a ramificação pode ser realmente útil.

- A ramificação é um método de seguir em uma nova direção ao criar um novo recurso e apenas combinar esse novo recurso com a parte principal do seu código, ou a ramificação principal, uma vez que você tenha terminado. Este fluxo de trabalho se parecerá mais com a imagem abaixo:

![Ramificação](https://cs50.harvard.edu/web/2020/notes/1/images/branch.png)

- A ramificação que você está visualizando no momento é determinada pelo `HEAD`, que aponta para uma das duas ramificações. Por padrão, o HEAD aponta para a ramificação master, mas também podemos verificar outras ramificações.
- Agora, vamos ver como implementamos a ramificação em nossos repositórios git:

  1. Execute `git branch` para ver em qual ramificação você está trabalhando no momento, que terá um asterisco à esquerda do seu nome.

  ![Terminal de Ramificação](https://cs50.harvard.edu/web/2020/notes/1/images/git_branch.png)

  2. Para criar uma nova ramificação, executaremos `git checkout -b <novo nome da ramificação>`

  ![Nova Ramificação](https://cs50.harvard.edu/web/2020/notes/1/images/new_branch.png)

  3. Alterne entre ramificações usando o comando `git checkout <nome da ramificação>` e faça commit de quaisquer alterações em cada ramificação.
  4. Quando estivermos prontos para mesclar nossas duas ramificações juntas, vamos verificar a ramificação que desejamos manter (quase sempre a ramificação master) e então executar o comando `git merge <nome da outra ramificação>`. Isso será tratado de forma semelhante a um push ou pull, e conflitos de mesclagem podem aparecer.

### Mais Recursos do GitHub <a id="more-github-features"></a>

Existem alguns recursos úteis específicos do GitHub que podem ajudar quando você está trabalhando em um projeto:

- **Forking**: Como usuário do GitHub, você tem a capacidade de _forkar_ qualquer repositório ao qual você tem acesso, o que cria uma cópia do repositório do qual você é proprietário. Fazemos isso clicando no botão "Fork" no canto superior direito.
- **Pull Requests**: Depois de fazer um fork de um repositório e fazer algumas alterações na sua versão, você pode querer solicitar que essas alterações sejam adicionadas à versão principal do repositório. Por exemplo, se você quisesse adicionar um novo recurso ao Bootstrap, você poderia forkar o repositório, fazer algumas alterações e então enviar um pull request. Esta solicitação de pull request pode então ser avaliada e possivelmente aceita pelas pessoas que gerenciam o repositório do Bootstrap. Esse processo de pessoas fazendo algumas edições e depois solicitando que sejam mescladas em um repositório principal é vital para o que é conhecido como _software de código aberto_, ou software criado por contribuições de vários desenvolvedores.
- **GitHub Pages**: O GitHub Pages é uma maneira simples de publicar um site estático na web. (Vamos aprender mais tarde sobre sites estáticos vs dinâmicos.) Para fazer isso:
  1. Crie um novo repositório no GitHub.
  2. Clone o repositório e faça alterações localmente, garantindo incluir um arquivo `index.html` que será a página principal do seu site.
  3. Faça o push dessas alterações para o GitHub.
  4. Acesse a página de configurações do seu repositório, role até o GitHub Pages, e escolha a ramificação master no menu suspenso.
  5. Role para baixo novamente para a parte de GitHub Pages da página de configurações e, após alguns minutos, você deve ver uma notificação de que "Seu site está publicado em: ..." incluindo um URL onde você pode encontrar seu site!
