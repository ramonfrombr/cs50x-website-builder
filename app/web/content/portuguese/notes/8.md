## Escalabilidade <a id="escalabilidade"></d>

Até agora, neste curso, construímos aplicações que são executadas apenas localmente em nossos computadores, mas eventualmente queremos lançar nossos sites para que possam ser acessados por qualquer pessoa na internet. Para fazer isso, executamos nossos sites em **servidores**, que são peças físicas de hardware dedicadas a executar aplicações. Os servidores podem ser locais (possuímos e mantemos servidores físicos onde nossa aplicação é hospedada) ou na nuvem (os servidores são de propriedade de uma empresa diferente, como Amazon ou Google, e pagamos pelo aluguel do espaço do servidor onde nossa aplicação é hospedada). Existem vantagens e desvantagens para ambas as opções:

-   **Personalização**: Hospedar seus próprios servidores oferece a capacidade de decidir exatamente como eles funcionam, permitindo mais flexibilidade do que a hospedagem baseada na nuvem.
-   **Expertise**: É muito mais simples hospedar uma aplicação na nuvem do que manter seus próprios servidores.
-   **Custo**: Como os sites de hospedagem de servidores precisam obter lucro, eles vão cobrar mais do que lhes custa manter seus servidores locais, tornando os servidores baseados na nuvem mais caros. No entanto, os custos iniciais de execução de servidores locais podem ser altos, pois é necessário comprar servidores físicos e potencialmente contratar alguém com a expertise para configurá-los.
-   **Escalabilidade**: A escalabilidade geralmente é mais fácil ao hospedar na nuvem. Por exemplo, se hospedarmos um site localmente que recebe 500 visitas por dia, e depois ele começa a receber 500.000 visitas por dia, teríamos que encomendar e configurar mais servidores físicos para lidar com as solicitações, e enquanto isso muitos de nossos usuários não serão capazes de acessar o site. A maioria dos sites de hospedagem na nuvem permitirá que você alugue espaço de servidor de forma flexível, pagando com base na quantidade de ações que seu site recebe.

Quando um usuário envia uma solicitação HTTP para este servidor, o servidor deve enviar uma resposta de volta. No entanto, na realidade, a maioria dos servidores recebe muito mais do que uma solicitação de cada vez, como mostra a imagem abaixo:

![servidor muitas entradas](https://cs50.harvard.edu/web/2020/notes/8/images/server0.png)

Aqui é onde encontramos o problema da escalabilidade. Um único servidor pode lidar apenas com um número limitado de solicitações de uma vez, nos obrigando a fazer planos sobre o que fazer quando nosso único servidor está sobrecarregado. Independentemente de decidirmos hospedar localmente ou na nuvem, temos que determinar quantas solicitações um servidor pode atender sem travar, o que pode ser feito usando uma série de ferramentas de **benchmarking**, incluindo o Apache Bench.

## Dimensionamento <a id="dimensionamento"></d>

Uma vez que tenhamos um limite superior para a quantidade de solicitações que nosso servidor pode lidar, podemos começar a pensar em como queremos lidar com o dimensionamento de nossa aplicação. Duas abordagens diferentes para o dimensionamento incluem:

1. **Dimensionamento Vertical**: No dimensionamento vertical, quando nosso servidor está sobrecarregado, simplesmente compramos ou construímos um servidor maior. No entanto, essa estratégia é limitada, pois há um limite superior para o quão poderoso um único servidor pode ser.
2. **Dimensionamento Horizontal**: No dimensionamento horizontal, quando nosso servidor está sobrecarregado, compramos ou construímos mais servidores e, em seguida, dividimos as solicitações entre nossos vários servidores.

## Balanceamento de Carga <a id="load-balancing"></a>

Quando usamos o escalonamento horizontal, nos deparamos com o problema adicional de como decidimos quais servidores são atribuídos a quais pedidos. Respondemos a essa pergunta empregando um **balanceador de carga**, que é outra peça de hardware que intercepta os pedidos recebidos e os atribui a um de nossos servidores. Existem várias métodos diferentes para decidir qual servidor recebe qual pedido, mas aqui estão alguns:

-   **Aleatório**: Neste método simples, o balanceador de carga decidirá aleatoriamente a qual servidor deve atribuir um pedido.
-   **Round-Robin**: Neste método, o balanceador de carga alternará qual servidor recebe um pedido recebido. Se tivermos três servidores, o primeiro pedido pode ir para o servidor A, o segundo para o servidor B, o terceiro para o servidor C e o quarto de volta para o servidor A.
-   **Menos Conexões**: Neste método, o balanceador de carga procura pelo servidor que está lidando com o menor número de pedidos no momento e atribui o pedido recebido a esse servidor. Isso nos permite garantir que não estamos sobrecarregando um servidor específico, mas também leva mais tempo para o balanceador de carga calcular o número de pedidos que cada servidor está lidando no momento do que para ele simplesmente escolher um servidor aleatório.

Não há um método de balanceamento de carga que seja estritamente melhor do que todos os outros métodos, e existem muitos métodos diferentes usados na prática. Um problema que pode surgir ao escalar horizontalmente é que podemos ter sessões armazenadas em um servidor, mas não em outro, e não queremos que os usuários tenham que reentrar informações apenas porque o balanceador de carga envia seu pedido para um novo servidor. Como muitos problemas de escalabilidade, existem várias abordagens para resolver o problema das sessões:

-   **Sessões Fixas**: Uma vez que um usuário visita um site, o balanceador de carga lembra para qual servidor ele foi enviado primeiro e garante que ele seja enviado para o mesmo. Uma grande preocupação com este método é que poderíamos acabar com um grande número de usuários fixados em um servidor, fazendo ele travar.
-   **Sessões de Banco de Dados**: Todas as sessões são armazenadas em um banco de dados ao qual todos os servidores têm acesso. Dessa forma, as informações do usuário estarão disponíveis, não importa para qual servidor ele seja designado. A desvantagem aqui é que leva tempo adicional e poder de computação para ler e escrever em um banco de dados.
-   **Sessões do Lado do Cliente**: Em vez de armazenar informações em nossos servidores, podemos optar por armazená-las localmente no navegador da web do usuário como cookies. As desvantagens deste método incluem a preocupação com a segurança de os usuários criarem cookies falsos que permitem que eles façam login como outro usuário e a preocupação computacional de enviar informações do cookie de um lado para o outro com cada pedido.

Assim como no balanceamento de carga, não há uma resposta definitiva para o problema das sessões, e o método que você escolher frequentemente dependerá de suas circunstâncias específicas.

## Dimensionamento automático <a id="autoscaling"></d>

Outro problema com o qual podemos nos deparar é que muitos sites são visitados com muito mais frequência em determinados momentos. Por exemplo, se decidirmos lançar nosso aplicativo "É Ano Novo?" de antes, podemos esperar que ele receba muito mais tráfego de fim de dezembro a início de janeiro do que em qualquer outra época do ano. Se comprarmos servidores suficientes para o site permanecer ativo durante o inverno, esses servidores ficariam ociosos pelo resto do ano, desperdiçando espaço e energia. Esse cenário deu origem à ideia de **dimensionamento automático**, que se tornou comum na computação em nuvem, onde o número de servidores utilizados pelo seu site pode aumentar e diminuir com base no número de solicitações que recebe. No entanto, o dimensionamento automático não é uma solução perfeita, pois leva tempo para determinar que um novo servidor é necessário e para lançar esse servidor. Outro problema potencial é que, quanto mais servidores estiverem em execução, maior será a oportunidade de um deles falhar.

### Falha do servidor <a id="server-failure"></d>

Ter vários servidores, no entanto, pode ajudar a evitar o que é conhecido como um **Ponto Único de Falha**, que é um componente de hardware que, ao falhar, fará com que o site inteiro falhe. Ao escalar horizontalmente, o balanceador de carga pode detectar quais servidores falharam ao enviar solicitações de **batimento cardíaco** periódicas para cada servidor, e depois parar de atribuir novas solicitações aos servidores que falharam. Neste momento, parece que simplesmente transferimos nosso ponto único de falha de um servidor para o balanceador de carga, mas podemos prever isso tendo balanceadores de carga de backup disponíveis caso o original venha a falhar.

## Escalonamento de Bancos de Dados <a id="escalando-bancos-de-dados"></d>

Além de escalarmos nossos servidores que processam requisições, também devemos pensar em maneiras de escalar nossos Bancos de Dados. Neste curso, utilizamos o SQLite que armazena dados em um arquivo no servidor. Conforme armazenamos mais e mais dados, às vezes faz mais sentido armazená-los em diferentes arquivos e talvez até em um servidor separado. Isso nos traz o problema de lidar com situações em que nosso servidor de banco de dados não consegue mais lidar com todas as requisições. Assim como em outras questões de escalabilidade, existem diversos métodos que podemos usar para mitigar esse problema:

-   **Particionamento Vertical**: Este é um método semelhante ao que usamos quando discutimos SQL pela primeira vez, onde dividimos nossos dados em várias tabelas diferentes em vez de ter informações redundantes em uma única tabela. (Sinta-se à vontade para rever a aula 4, onde dividimos a tabela `flights` em uma tabela `flights` e uma tabela `airports`).
-   **Particionamento Horizontal**: Este método envolve armazenar várias tabelas com o mesmo formato, mas com informações diferentes. Por exemplo, poderíamos dividir uma tabela `flights` em uma tabela `voos_domesticos` e uma tabela `voos_internacionais`. Dessa forma, quando desejarmos procurar por um voo de JFK para LHR, não precisamos perder tempo procurando em uma tabela cheia de voos domésticos. Uma desvantagem deste método é que pode ser caro unir várias tabelas depois que elas foram divididas.

### Replicação de Banco de Dados <a id="replicacao-de-banco-de-dados"></d>

Mesmo depois de escalarmos um banco de dados, ainda enfrentamos um único ponto de falha. Se nosso servidor de banco de dados falhar, todos os nossos dados podem ser perdidos. Assim como adicionamos mais servidores para evitar um único ponto de falha, podemos adicionar cópias do nosso banco de dados para garantir que a falha de um banco de dados não desligue nossa aplicação. Da mesma forma, existem métodos diferentes de replicação de banco de dados, dos quais dois dos mais populares são:

-   **Replicação de Único Primário**: Neste método, existem vários bancos de dados, mas apenas um deles é considerado o banco de dados primário, o que significa que você pode ler e escrever em um dos bancos de dados, mas apenas ler dos outros. Quando o banco de dados principal é atualizado, os outros bancos de dados são então atualizados para corresponder ao banco de dados principal. Uma desvantagem deste método é que ele ainda contém um único ponto de falha quando se trata de escrever no banco de dados.

![visual de único primário](https://cs50.harvard.edu/web/2020/notes/8/images/single_primary.png)

-   **Replicação de Múltiplos Primários**: Neste método, todos os bancos de dados podem ser lidos e escritos. Isso resolve o problema do único ponto de falha, mas traz um trade-off: agora é muito mais difícil manter todos os bancos de dados atualizados, pois cada banco de dados deve estar ciente das alterações em todos os outros bancos de dados. Esse sistema também nos coloca na possibilidade de alguns conflitos:
    -   **Conflito de Atualização**: Com vários bancos de dados, um usuário pode tentar editar uma linha em um banco de dados, enquanto outro usuário tenta editar essa mesma linha em um banco de dados diferente, causando um problema quando os bancos de dados são sincronizados.
    -   **Conflito de Unicidade**: Cada linha em um banco de dados SQL deve ter um identificador único, e podemos ter o problema de atribuir o mesmo ID a duas entradas diferentes em dois bancos de dados diferentes.
    -   **Conflito de Exclusão**: Um usuário pode excluir uma linha enquanto outro usuário tenta atualizá-la.

![visual de múltiplos primários](https://cs50.harvard.edu/web/2020/notes/8/images/multi_primary.png)

## Armazenamento em Cache <a id="caching"></d>

Sempre que estamos trabalhando com bancos de dados maiores, é importante reconhecer que cada interação que temos com um banco de dados é custosa. Portanto, desejamos minimizar o número de chamadas feitas ao nosso servidor de banco de dados. Vamos olhar, por exemplo, para o site do [New York Times](https://www.nytimes.com/). O New York Times pode ter um banco de dados com todos os seus artigos que é consultado e algum modelo que é renderizado toda vez que alguém carrega a página inicial, mas isso seria um desperdício de recursos, pois os artigos exibidos na página inicial provavelmente não mudam muito de segundo a segundo. Uma forma de lidar com esse problema é usando o **Armazenamento em Cache**, que é a ideia de armazenar algumas informações em um local mais acessível se anteciparmos a necessidade delas novamente em um futuro próximo.

Uma forma de implementar o cache é armazenando dados no navegador da web do usuário, para que quando um usuário carrega determinadas páginas, nenhuma solicitação ao servidor precise ser feita. Uma forma de fazer isso é incluir esta linha no cabeçalho de uma resposta HTTP:

    Cache-Control: max-age=86400

Isso informará ao navegador que, ao visitar uma página, desde que eu tenha visitado essa página nos últimos 86400 milissegundos, nenhuma solicitação precisa ser feita ao servidor. Este método é comumente usado por navegadores da web, especialmente com arquivos que têm menos probabilidade de mudar em curtos períodos, como um arquivo CSS. Para ter mais controle sobre esse processo, também podemos adicionar um `ETag` ao cabeçalho da resposta HTTP, que é uma sequência única de caracteres que representa uma versão específica de um documento. Isso é útil porque solicitações futuras podem incluir essa tag e compará-la com a tag do último documento no servidor, retornando um documento inteiro somente quando os dois forem diferentes.

Além do armazenamento em cache do lado do cliente discutido anteriormente, muitas vezes é útil incluir um cache no lado do servidor. Com esse cache, nossa configuração backend terá um aspecto semelhante ao mostrado abaixo, em que todos os servidores têm acesso a um cache.

![armazenamento em cache no servidor](https://cs50.harvard.edu/web/2020/notes/8/images/server_cache.png)

O Django fornece seu próprio [framework de cache](https://docs.djangoproject.com/en/4.0/topics/cache/) que nos permitirá incorporar o armazenamento em cache em nossos projetos. Este framework oferece várias formas de implementar um cache:

-   **Armazenamento em Cache por Visualização**: Isso nos permite decidir que, uma vez que uma visualização específica foi carregada, essa mesma visualização pode ser renderizada sem passar pela função durante o próximo período de tempo especificado.
-   **Armazenamento de Fragmentos de Template**: Este cache fragmenta partes específicas de um template para que não precisem ser re-renderizadas. Por exemplo, podemos ter uma barra de navegação que raramente muda, o que significa que poderíamos economizar tempo não recarregando-a.
-   **API de Cache de Baixo Nível**: Isso permite fazer um armazenamento em cache mais flexível, essencialmente armazenando qualquer informação desejada.

Não entraremos em detalhes aqui sobre como implementar o armazenamento em cache no Django, mas dê uma olhada na [documentação](https://docs.djangoproject.com/en/4.0/topics/cache/) se estiver interessado!

## Segurança <a id="seguranca"></d>

Agora, vamos começar a discutir como garantir que nossas aplicações web sejam seguras, o que envolverá muitas medidas diferentes que abrangem quase todos os tópicos que discutimos neste curso.

### Git e GitHub <a id="git-e-github"></d>

Uma das maiores vantagens do Git e GitHub é o quão fácil eles tornam compartilhar e contribuir para **softwares de código aberto**, que podem ser vistos e contribuídos por qualquer pessoa na internet. Uma desvantagem disso é que se, a qualquer momento, você cometer um arquivo que inclua algumas credenciais privadas como uma senha ou chave de API, essas credenciais podem estar publicamente disponíveis.

## HTML <a id="html"></d>

Existem muitas vulnerabilidades que surgem ao usar HTML. Uma fraqueza comum é conhecida como **Ataque de Phishing**, que ocorre quando um usuário que pensa que está indo para uma página é na verdade levado para outra. Estes são problemas que não necessariamente podemos prever ao projetar um website, mas definitivamente devemos mantê-los em mente ao interagir com a web. Por exemplo, um usuário malicioso pode escrever este HTML:

    <!DOCTYPE html>
    <html lang="pt">
        <head>
            <title>Link</title>
        </head>
        <body>
            <a href="https://cs50.harvard.edu/">https://www.google.com/</a>
        </body>
    </html>

Que funciona da seguinte forma:

![ataque de phishing](https://cs50.harvard.edu/web/2020/notes/8/images/phishing.gif)

O fato de que o HTML é realmente enviado a um usuário como parte de uma solicitação abre mais vulnerabilidades, pois todos têm acesso ao layout e estilo que permitiram criar seu site. Por exemplo, um hacker poderia ir para [bankofamerica.com](https://cs50.harvard.edu/), copiar todo o HTML deles e colá-lo em seu próprio site, criando um site que parece exatamente com o do Bank of America. O hacker poderia então redirecionar o formulário de login na página para que todos os nomes de usuário e senhas sejam enviados para ele. (Também, aqui está o [link real do Bank of America](https://www.bankofamerica.com/) - apenas para ver se você está verificando os URLs antes de clicar!)

## HTTPS <a id="https"></d>

Conforme discutimos anteriormente no curso, a maioria das interações que ocorrem online segue o protocolo HTTP, embora cada vez mais transações usem HTTPS, que é uma versão criptografada do HTTP. Enquanto utilizam esses protocolos, as informações são transferidas de um computador para outro por meio de uma série de servidores, conforme ilustrado abaixo.

![transferindo](https://cs50.harvard.edu/web/2020/notes/8/images/servers.png)

Muitas vezes, não há como garantir que todas essas transferências sejam seguras, portanto, é importante que todas essas informações transferidas sejam **criptografadas**, significando que os caracteres da mensagem são alterados para que o remetente e o destinatário da mensagem possam entendê-la, mas ninguém mais possa.

### Criptografia de Chave Secreta <a id="secret-key-cryptography"></d>

Uma abordagem para isso é conhecida como **Criptografia de Chave Secreta**. Neste método, o remetente e o destinatário têm acesso a uma chave secreta que somente eles conhecem. Em seguida, a chave secreta é usada pelo remetente para criptografar uma mensagem que é então enviada para o destinatário, que usa a chave secreta para descriptografar a mensagem. Este método é extremamente seguro, mas apresenta um grande problema em termos de praticidade. Para funcionar, tanto o remetente quanto o destinatário devem ter acesso à chave secreta, o que significa que eles devem se encontrar pessoalmente para trocar a chave com segurança. Com o número de diferentes websites com os quais interagimos diariamente, é evidente que os encontros pessoais não são uma opção.

### Criptografia de Chave Pública <a id="public-key-cryptography"></d>

Um avanço incrível na criptografia que permite que a internet funcione como faz hoje é conhecido como **Criptografia de Chave Pública**. Neste método, existem duas chaves: uma é pública e pode ser compartilhada, e a outra deve ser mantida privada. Uma vez que essas chaves são estabelecidas (existem vários métodos matemáticos diferentes para criar pares de chaves, o que poderia ser um curso completo só para isso, então não vamos discuti-los aqui), um remetente pode procurar a chave pública de um destinatário e usá-la para criptografar uma mensagem, e então o destinatário pode usar sua chave privada para descriptografar a mensagem. Quando utilizamos HTTPS em vez de HTTP, sabemos que nossa solicitação está sendo protegida usando criptografia de chave pública.

## Bancos de dados <a id="bancos-de-dados"></d>

Além de nossas requisições e respostas, também devemos garantir que nossos bancos de dados sejam seguros. Uma coisa comum que precisaremos armazenar são as informações do usuário, incluindo nomes de usuário e senhas conforme a tabela abaixo:

![tabela ruim](https://cs50.harvard.edu/web/2020/notes/8/images/passwords.png)

No entanto, nunca devemos armazenar senhas em texto simples caso uma pessoa não autorizada obtenha acesso ao nosso banco de dados. Em vez disso, devemos utilizar uma **função de hash**, uma função que recebe um texto como entrada e produz uma sequência aparentemente aleatória, para criar um hash de cada senha, como mostrado na tabela abaixo:

![tabela boa](https://cs50.harvard.edu/web/2020/notes/8/images/hashes.png)

É importante observar que uma função de hash é **unidirecional**, ou seja, ela pode converter uma senha em um hash, mas não pode reverter um hash em uma senha. Isso significa que qualquer empresa que armazene informações de usuários dessa forma na realidade não conhece nenhuma das senhas dos usuários, o que implica que cada vez que um usuário tenta fazer login, a senha inserida será transformada em um hash e comparada com o hash existente. Felizmente, esse processo já é controlado pelo Django. Uma implicação dessa técnica de armazenamento é que, quando um usuário esquece sua senha, a empresa não tem como informar qual era a senha antiga, o que significa que o usuário terá que criar uma nova.

Existem casos em que o desenvolvedor terá que decidir quanto de informação está disposto a vazar. Por exemplo, muitos sites possuem uma página para recuperação de senhas que se parece com esta:

![esqueceu sua senha?](https://cs50.harvard.edu/web/2020/notes/8/images/forgot0.png)

Como desenvolvedor, você pode querer incluir uma mensagem de sucesso ou de erro após o envio:

![mensagem de sucesso](https://cs50.harvard.edu/web/2020/notes/8/images/sent.png)

![mensagem de erro](https://cs50.harvard.edu/web/2020/notes/8/images/error.png)

Mas perceba como, digitando e-mails, qualquer pessoa poderia determinar quem possui um e-mail registrado naquele site. Isso pode ser completamente aceitável em casos em que é irrelevante se uma pessoa usa o site ou não (talvez o Facebook), mas extremamente imprudente se o fato de você ser membro de um determinado site puder te colocar em perigo (talvez em um grupo de apoio online para vítimas de abuso).

Outra forma de vazamento de dados é no tempo que leva para obter uma resposta. Provavelmente leva menos tempo para rejeitar alguém com um e-mail inválido do que uma pessoa com um endereço de e-mail correto e uma senha incorreta.

Como discutimos anteriormente no curso, devemos estar atentos aos Ataques de Injeção SQL sempre que utilizamos queries SQL diretas em nosso código.

### APIs <a id="apis"></d>

Frequentemente utilizamos JavaScript em conjunto com APIs para construir aplicações de página única. No caso de construirmos nossa própria API, existem alguns métodos que podemos utilizar para manter nossa API segura:

-   **Chaves de API**: Processar apenas solicitações de clientes da API que possuem uma chave fornecida por nós.
-   **Limitação de Taxa**: Limitar o número de solicitações que um usuário pode fazer dentro de um determinado período de tempo. Isso ajuda a proteger contra **Ataques de Negação de Serviço (DOS)**, nos quais um usuário mal-intencionado faz tantas chamadas para sua API que ela trava.
-   **Autenticação de Rota**: Existem muitos casos em que não queremos dar acesso a todos os nossos dados, então podemos usar autenticação de rota para garantir que apenas usuários específicos possam ver dados específicos.

### Variáveis de Ambiente <a id="variaveis-de-ambiente"></d>

Assim como queremos evitar armazenar senhas em texto simples, também queremos evitar incluir chaves de API em nosso código-fonte. Uma forma comum de evitar isso é utilizando as **variáveis de ambiente**, ou variáveis armazenadas no ambiente do seu sistema operacional ou servidor. Dessa forma, em vez de incluir uma string de texto em nosso código-fonte, podemos incluir uma referência a uma variável de ambiente.

## JavaScript <a id="javascript"></a>

Existem alguns tipos de ataques que usuários maliciosos podem tentar usando JavaScript. Um exemplo é conhecido como **Cross-Site Scripting**, que é quando um usuário escreve seu próprio código JavaScript e o executa em seu site. Por exemplo, vamos imaginar que temos uma aplicação Django com uma única URL:

```python
urlpatterns = [
    path("<path:path>", views.index, name="index")
]
```

e uma única view:

    def index(request, path):
        return HttpResponse(f"Caminho Solicitado: {path}")

Este site basicamente informa ao usuário qual URL eles navegaram:

![bom caminho](https://cs50.harvard.edu/web/2020/notes/8/images/pathworks.png)

Mas um usuário agora pode facilmente inserir algum JavaScript na página digitando-o na URL:

![caminho ruim](https://cs50.harvard.edu/web/2020/notes/8/images/inject.gif)

Embora este exemplo de `alert` seja relativamente inofensivo, não seria muito mais difícil incluir um JavaScript que manipula o DOM ou usa `fetch` para enviar uma requisição.

### Cross-Site Request Forgery <a id="cross-site-request-forgery"></a>

Já discutimos como podemos usar o Django para prevenir ataques CSRF, mas vamos ver o que poderia acontecer sem essa proteção. Como exemplo, imagine que um banco tenha uma URL que você poderia visitar para transferir dinheiro de sua conta. Uma pessoa poderia facilmente criar um link que faria essa transferência:

    <a href="http://seubanco.com/transferir?para=brian&valor=2800">
        Clique Aqui!
    </a>

Esse ataque pode ser ainda mais sutil do que um link. Se a URL for colocada em uma imagem, ela será acessada conforme seu navegador tenta carregar a imagem:

     <img src="http://seubanco.com/transferir?para=brian&valor=2800">

Por causa disso, sempre que estiver desenvolvendo uma aplicação que pode aceitar alguma mudança de estado, ela deve ser feita usando uma requisição POST. Mesmo que o banco exija uma requisição POST, campos de formulário ocultos ainda podem enganar os usuários para enviar uma solicitação acidentalmente. O formulário a seguir nem sequer espera o usuário clicar; ele envia automaticamente!

```xml
<body onload="document.forms[0].submit()">
    <form action="https://seubanco.com/transferir"
    method="post">
        <input type="hidden" name="para" value="brian">
        <input type="hidden" name="amount" value="2800">
        <input type="submit" value="Clique Aqui!">
    </form>
</body>
```

O exemplo acima é do que **Cross-Site Request Forgery** pode parecer. Podemos impedir ataques como esses criando um token CSRF ao carregar uma página da web e, em seguida, aceitando apenas formulários com um token válido.
