## SQL <a id="sql"></a>

[SQL](https://www.w3schools.com/sql/), ou Structured Query Language, é uma linguagem de programação que nos permite atualizar e consultar bancos de dados.

![Logotipo do SQL](https://cs50.harvard.edu/web/2020/notes/4/images/sql.png)

### Bancos de Dados <a id="databases"></a>

Antes de aprendermos como usar a linguagem SQL, devemos discutir como nossos dados são armazenados. Ao usar SQL, trabalharemos com um [banco de dados relacional](https://www.oracle.com/database/what-is-a-relational-database/#:~:text=A%20relational%20database%20is%20a,of%20representing%20data%20in%20tables.), onde podemos encontrar todos os nossos dados armazenados em várias [tabelas](https://www.essentialsql.com/what-is-a-database-table/). Cada uma dessas tabelas é composta por um número fixo de colunas e um número flexível de linhas.

Para ilustrar como trabalhar com SQL, usaremos o exemplo de um site para uma companhia aérea usado para controlar voos e passageiros. Na tabela a seguir, vemos que estamos controlando vários voos, cada um dos quais tem uma `origem`, um `destino` e uma `duração`.

![Voos 0](https://cs50.harvard.edu/web/2020/notes/4/images/flights0.png)

Existem diversos sistemas de gerenciamento de banco de dados relacionais comumente usados para armazenar informações e que podem interagir facilmente com comandos SQL:

- [MySQL](https://www.mysql.com/)
- [PostgreSQL](https://www.postgresql.org/)
- [SQLite](https://www.sqlite.org/index.html)
- ...

Os dois primeiros, MySQL e PostgreSQL, são sistemas de gerenciamento de banco de dados mais robustos que geralmente são executados em servidores separados daqueles que executam um site. SQLite, por outro lado, é um sistema mais leve que pode armazenar todos os seus dados em um único arquivo. Usaremos o SQLite ao longo deste curso, pois é o sistema padrão usado pelo Django.

### Tipos de Colunas <a id="column-types"></a>

Assim como trabalhamos com vários tipos de variáveis em Python, o SQLite possui [tipos](https://www.sqlite.org/datatype3.html) que representam diferentes formas de informações. Outros sistemas de gerenciamento podem ter tipos de dados diferentes, mas todos são bastante semelhantes aos do SQLite:

- `TEXT`: Para strings de texto (Por exemplo, o nome de uma pessoa)
- `NUMERIC`: Uma forma mais genérica de dados numéricos (Por exemplo, uma data ou um valor booleano)
- `INTEGER`: Qualquer número sem decimais (Por exemplo, a idade de uma pessoa)
- `REAL`: Qualquer número real (Por exemplo, o peso de uma pessoa)
- `BLOB` (Binary Large Object): Qualquer outro dado binário que possamos querer armazenar em nosso banco de dados (Por exemplo, uma imagem)

## Tabelas <a id="tabelas"></a>

Agora, para começar a usar o SQL para interagir com um banco de dados, vamos começar criando uma nova tabela. O [comando para criar uma nova tabela](https://www.w3schools.com/sql/sql_create_table.asp) se parece com isso:

    CREATE TABLE voos(
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        origem TEXT NOT NULL,
        destino TEXT NOT NULL,
        duração INTEGER NOT NULL
    );

No comando acima, estamos criando uma nova tabela que decidimos chamar de `voos`, e adicionamos quatro colunas a esta tabela:

1. `id`: É frequentemente útil ter um número que nos permite identificar unicamente cada linha em uma tabela. Aqui especificamos que `id` é um número inteiro, e também que é nossa [chave primária](https://www.w3schools.com/sql/sql_primarykey.ASP), o que significa que é nosso identificador único. Além disso, especificamos que ele `AUTOINCREMENT`, o que significa que não precisaremos fornecer um id toda vez que adicionarmos à tabela, pois será feito automaticamente.
2. `origem`: Aqui especificamos que isso será um campo de texto e, ao escrever `NOT NULL`, exigimos que ele tenha um valor.
3. `destino`: Novamente, especificamos que isso será um campo de texto e impedimos que seja nulo.
4. `duração`: Novamente, esse valor não pode ser nulo, mas desta vez é representado por um número inteiro em vez de texto.

Acabamos de ver a restrição `NOT NULL` e `PRIMARY KEY` ao criar uma coluna, mas existem várias outras [restrições](https://www.tutorialspoint.com/sqlite/sqlite_constraints.htm) disponíveis para nós:

- `CHECK`: Garante que certas restrições sejam atendidas antes de permitir que uma linha seja adicionada/modificada.
- `DEFAULT`: Fornece um valor padrão se nenhum valor for fornecido.
- `NOT NULL`: Garante que um valor seja fornecido.
- `PRIMARY KEY`: Indica que esta é a maneira principal de pesquisar por uma linha no banco de dados.
- `UNIQUE`: Garante que nenhuma duas linhas tenham o mesmo valor nessa coluna.
- ...

Agora que vimos como criar uma tabela, vamos ver como podemos adicionar linhas a ela. No SQL, fazemos isso usando o comando `INSERT`:

    INSERT INTO voos
        (origem, destino, duração)
        VALUES ("Nova Iorque", "Londres", 415);

No comando acima, especificamos o nome da tabela que desejamos inserir, em seguida, fornecemos uma lista dos nomes das colunas nas quais forneceremos informações e, em seguida, especificamos os `VALORES` que gostaríamos de preencher nessa linha na tabela, garantindo que os `VALORES` estejam na mesma ordem que nossa lista correspondente de colunas. Observe que não precisamos fornecer um valor para `id` porque ele está incrementando automaticamente.

## SELECT

Uma vez que uma tabela tenha sido populada com algumas linhas, provavelmente queremos uma maneira de acessar os dados dentro dessa tabela. Fazemos isso usando a consulta [SELECT](https://www.w3schools.com/sql/sql_select.asp) do SQL. A consulta `SELECT` mais simples em nossa tabela de voos pode ser algo como:

    SELECT * FROM flights;

O comando acima (\*) recupera todos os dados de nossa tabela de voos

![all](https://cs50.harvard.edu/web/2020/notes/4/images/all.png)

Pode ser o caso de que não precisamos realmente de todas as colunas do banco de dados, apenas a origem e o destino. Para acessar apenas essas colunas, podemos substituir o \* pelos nomes das colunas às quais gostaríamos de acessar. A consulta a seguir retorna todas as origens e destinos.

    SELECT origin, destination FROM flights;

![Só duas colunas](https://cs50.harvard.edu/web/2020/notes/4/images/flights1.png)

À medida que nossas tabelas ficam maiores, queremos também restringir quais linhas nossa consulta retorna. Fazemos isso adicionando um [WHERE](https://www.w3schools.com/sql/sql_where.asp) seguido por alguma condição. Por exemplo, o seguinte comando seleciona apenas a linha com um `id` de `3`:

    SELECT * FROM flights WHERE id = 3;

![apenas uma linha](https://cs50.harvard.edu/web/2020/notes/4/images/where0.png)

Podemos filtrar por qualquer coluna, não apenas por `id`!

    SELECT * FROM flights WHERE origin = "Nova Iorque";

![Origem é Nova Iorque](https://cs50.harvard.edu/web/2020/notes/4/images/where1.png)

Também podemos usar mais do que apenas igualdade para filtrar nossos voos. Para valores inteiros e reais, podemos usar maior que ou menor que:

    SELECT * FROM flights WHERE duration > 500;

E também podemos usar outras lógicas ([AND, OR](https://www.w3schools.com/sql/sql_and_or.asp)) como no Python:

    SELECT * FROM flights WHERE duration > 500 AND destination = "Paris";

Também podemos usar a palavra-chave [IN](https://www.w3schools.com/sql/sql_in.asp) para verificar se um dado é uma das várias opções:

### UPDATE

Agora vimos como adicionar e pesquisar tabelas, mas também podemos querer atualizar linhas de uma tabela que já existem. Fazemos isso utilizando o comando [UPDATE](https://www.w3schools.com/sql/sql_update.asp) como mostrado abaixo. ## Juntando Tabelas <a id="joining-tables"></a>

Até agora, só trabalhamos com uma tabela por vez, mas muitos bancos de dados na prática são populados por várias tabelas que se relacionam de alguma forma. No nosso exemplo de voos, vamos imaginar que também queremos adicionar um código de aeroporto para acompanhar a cidade. Da forma como nossa tabela está configurada atualmente, teríamos que adicionar mais duas colunas para cada linha. Além disso, estaríamos repetindo informações, já que teríamos que escrever em vários lugares que a cidade X está associada ao código Y.

Uma forma de resolver esse problema é decidir ter uma tabela que mantenha o controle dos voos e, em seguida, outra tabela para manter o controle dos aeroportos. A segunda tabela pode se parecer com isso:

![Tabela de Aeroportos](https://cs50.harvard.edu/web/2020/notes/4/images/airports.png)

Agora temos uma tabela relacionando códigos e cidades, em vez de armazenar um nome completo de cidade na nossa tabela de voos, economizando espaço de armazenamento se pudermos apenas salvar o `id` desse aeroporto. Portanto, devemos reescrever a tabela de voos de acordo. Como estamos usando a coluna `id` da tabela de aeroportos para popular `origin_id` e `destination_id`, chamamos esses valores de [Chaves Estrangeiras](https://www.w3schools.com/sql/sql_foreignkey.asp).

![Novos Voos](https://cs50.harvard.edu/web/2020/notes/4/images/flights2.png)

Além de voos e aeroportos, uma companhia aérea também pode querer armazenar dados sobre seus passageiros, como em qual voo cada passageiro estará. Usando o poder dos bancos de dados relacionais, podemos adicionar outra tabela que armazena nomes completos e uma chave estrangeira representando o voo em que estão.

![Tabela Simples de Passageiros](https://cs50.harvard.edu/web/2020/notes/4/images/simple_pass.png)

Podemos melhorar ainda mais, pois a mesma pessoa pode estar em mais de um voo. Para lidar com isso, podemos criar uma tabela `people` que armazena nomes completos e uma tabela `passengers` que associa pessoas a voos.

![Pessoas](https://cs50.harvard.edu/web/2020/notes/4/images/people.png)

![Passageiros](https://cs50.harvard.edu/web/2020/notes/4/images/passengers.png)

Como, neste caso, uma única pessoa pode estar em vários voos e um único voo pode ter várias pessoas, chamamos a relação entre `voos` e `pessoas` de uma relação **Muitos para Muitos**. A tabela `passageiros` que conecta as duas é conhecida como uma **tabela de associação**.

### Consulta JOIN <a id="join-query"></a>

Embora nossos dados agora estejam armazenados de forma mais eficiente, parece que pode ser mais difícil consultar nossos dados. Felizmente, o SQL conta com uma consulta [JOIN](https://www.w3schools.com/sql/sql_join.asp) onde podemos combinar duas tabelas para os propósitos de outra consulta.

Por exemplo, digamos que queremos encontrar a origem, destino e primeiro nome de cada viagem que um passageiro está fazendo. Também, para simplificar nesta tabela, vamos estar usando a tabela de `passageiros` não otimizada que inclui o id do voo, primeiro nome e sobrenome. A primeira parte desta consulta parece bastante familiar:

    SELECIONAR first, origin, destination
    DE ...

Mas encontramos um problema aqui porque `first` está armazenado na tabela de `passageiros`, enquanto `origin` e `destination` estão armazenadas na tabela de `voos`. Resolvemos isso juntando as duas tabelas usando o fato de que `flight_id` na tabela de `passageiros` corresponde a `id` na tabela de `voos`:

    SELECIONAR first, origin, destination
    DE voos JOIN passageiros
    ON passageiros.flight_id = voos.id;

![Join embaçado](https://cs50.harvard.edu/web/2020/notes/4/images/join.png)

Acabamos de utilizar algo chamado [JOIN INTERNO](https://www.w3schools.com/sql/sql_join_inner.asp), o que significa que estamos ignorando linhas que não têm correspondências entre as tabelas, mas existem outros tipos de joins, incluindo [**LEFT Joins**](https://www.w3schools.com/sql/sql_join_left.asp), [**RIGHT Joins**](https://www.w3schools.com/sql/sql_join_right.asp) e [**FULL OUTER Joins**](https://www.w3schools.com/sql/sql_join_full.asp), os quais não discutiremos em detalhes aqui.

### Indexação <a id="indexing"></a>

Uma maneira de tornar nossas consultas mais eficientes ao lidar com tabelas grandes é criar um índice semelhante ao índice que você pode ver no final de um livro. Por exemplo, se sabemos que frequentemente procuraremos passageiros pelo sobrenome, poderíamos criar um índice do sobrenome para o id usando o comando:

    CRIAR ÍNDICE name_index EM passageiros (last);

### Vulnerabilidades do SQL <a id="sql-vulnerabilities"></a>

Agora que sabemos o básico de como usar SQL para trabalhar com dados, é importante apontar as principais vulnerabilidades associadas ao uso de SQL. Vamos começar com [Injeção de SQL](https://www.w3schools.com/sql/sql_injection.asp).

Um ataque de injeção de SQL ocorre quando um usuário mal-intencionado insere código SQL como entrada em um site para burlar as medidas de segurança do site. Por exemplo, digamos que temos uma tabela armazenando nomes de usuário e senhas, e então um formulário de login na página inicial de um site. Podemos procurar pelo usuário usando uma consulta como esta:

    SELECIONAR * DE users
    ONDE nome de usuário = nome de usuário E senha = senha;

Um usuário chamado Harry pode ir para este site e digitar `harry` como nome de usuário e `12345` como senha, caso em que a consulta ficaria assim:

    SELECIONAR * DE users
    ONDE nome de usuário = "harry" E senha = "12345";

Um hacker, por outro lado, pode digitar `harry" --` como nome de usuário e nada como senha. Acontece que `--` significa um comentário em SQL, o que significa que a consulta ficaria assim:

    SELECIONAR * DE users
    ONDE nome de usuário = "harry"--" E senha = "12345";

Como nessa consulta a verificação de senha foi comentada, o hacker pode fazer login na conta de Harry sem saber a senha dele. Para resolver esse problema, podemos usar:

- Caracteres de escape para garantir que o SQL trate a entrada como texto simples e não como código SQL.
- Uma camada de abstração em cima do SQL que inclui sua própria sequência de escape, para que não precisemos escrever consultas SQL nós mesmos.

A outra principal vulnerabilidade quando se trata de SQL é conhecida como [Condição de Corrida (Race Condition)](https://searchstorage.techtarget.com/definition/race-condition#:~:text=A%20race%20condition%20is%20an,sequence%20to%20be%20done%20correctly.).

Uma condição de corrida é uma situação que ocorre quando várias consultas a um banco de dados ocorrem simultaneamente. Quando essas não são manipuladas adequadamente, podem surgir problemas nos momentos precisos em que os bancos de dados são atualizados. Por exemplo, digamos que eu tenha $150 na minha conta bancária. Uma condição de corrida pode ocorrer se eu fizer login na minha conta bancária tanto no meu celular quanto no meu laptop e tentar sacar $100 em cada dispositivo. Se os desenvolvedores de software do banco não lidaram corretamente com as condições de corrida, posso conseguir sacar $200 de uma conta que tem apenas $150. Uma solução potencial para esse problema seria bloquear o banco de dados. Não permitiríamos nenhuma outra interação com o banco de dados até que uma transação seja concluída. No exemplo do banco, após clicar em navegar para a página "Fazer um Saque" no meu computador, o banco pode não permitir que eu acesse essa página no meu celular.

## Modelos do Django <a id="django-models"></a>

Os [Modelos do Django](https://docs.djangoproject.com/pt-br/4.0/topics/db/models/) são um nível de [abstração](https://techterms.com/definition/abstraction) em cima do SQL que nos permite trabalhar com bancos de dados usando classes e objetos Python em vez de consultas SQL diretas.

Vamos começar a usar modelos criando um projeto Django para nossa companhia aérea e criando um aplicativo dentro desse projeto.

    django-admin startproject airline
    cd airline
    python manage.py startapp flights

Agora precisaremos seguir o processo de adicionar um aplicativo como de costume:

1.  Adicione `flights` à lista `INSTALLED_APPS` em `settings.py`
2.  Adicione uma rota para `flights` em `urls.py`:

         path("flights/", include("flights.urls")),

3.  Crie um arquivo `urls.py` dentro do aplicativo `flights`. E preencha-o com importações e listas padrão de `urls.py`.

Agora, em vez de criar caminhos reais e começar em `views.py`, criaremos alguns modelos no arquivo `models.py`. Neste arquivo, delimitaremos os dados que desejamos armazenar em nossa aplicação. Em seguida, o Django determinará a sintaxe SQL necessária para armazenar informações em cada um de nossos modelos. Vamos dar uma olhada em como um modelo para um voo único pode ser:

    class Flight(models.Model):
        origin = models.CharField(max_length=64)
        destination = models.CharField(max_length=64)
        duration = models.IntegerField()

Vamos analisar o que está acontecendo nessa definição de modelo:

- Na primeira linha, criamos um novo modelo que **estende** a classe model do Django.
- Abaixo, adicionamos os campos para origem, destino e duração. Os dois primeiros são [Campos de Char](https://docs.djangoproject.com/pt-br/4.0/ref/forms/fields/#charfield), o que significa que armazenam strings, e o terceiro é um [Campo de Inteiro](https://docs.djangoproject.com/pt-br/4.0/ref/forms/fields/#integerfield). Estes são apenas dois dos muitos [Campos de Campo Incorporados do Django](https://docs.djangoproject.com/pt-br/4.0/ref/forms/fields/#built-in-field-classes).
- Especificamos comprimentos máximos de 64 para os dois Campos de Caracteres. Você pode verificar as especificações disponíveis para um determinado campo verificando a [documentação](https://docs.djangoproject.com/pt-br/4.0/ref/forms/fields/#built-in-field-classes).

    class Airport(models.Model):

        destination = models.ForeignKey(Airport, on_delete=models.CASCADE, related_name="chegadas")
        duration = models.IntegerField()

        def __str__(self):
            return f"{self.id}: {self.origin} para {self.destination}"

Já vimos tudo em nossa nova classe `Aeroporto` antes, mas as mudanças nos campos `origem` e `destino` dentro da classe `Voo` são novas para nós:

- Especificamos que os campos `origem` e `destino` são cada um [Chaves Estrangeiras](https://docs.djangoproject.com/en/4.0/topics/db/examples/many_to_one/), o que significa que se referem a outro objeto.
- Ao introduzir `Aeroporto` como nosso primeiro argumento, estamos especificando o tipo de objeto a que este campo se refere.
- O próximo argumento, `on_delete=models.CASCADE`, fornece instruções sobre o que deve acontecer se um aeroporto for excluído. Neste caso, especificamos que quando um aeroporto é excluído, todos os voos associados a ele também devem ser excluídos. Existem [várias outras opções](https://docs.djangoproject.com/en/4.0/ref/models/fields/#django.db.models.ForeignKey.on_delete) além de `CASCADE`.
- Fornecemos um [nome relacionado](https://docs.djangoproject.com/en/4.0/ref/models/fields/#django.db.models.ForeignKey.related_name), que nos dá uma maneira de pesquisar todos os voos com um determinado aeroporto como origem ou destino.

## Relacionamentos Muitos-para-Muitos <a id="many-to-many-relationships"></a>

Agora, vamos trabalhar na integração de passageiros em nossos modelos. Vamos criar um modelo de passageiro para começar:

```python
class Passenger(models.Model):
    first = models.CharField(max_length=64)
    last = models.CharField(max_length=64)
    flights = models.ManyToManyField(Flight, blank=True, related_name="passengers")

    def __str__(self):
        return f"{self.first} {self.last}"
```

- Como discutimos, os passageiros têm um relacionamento **Muitos para Muitos** com voos, que descrevemos no Django usando o ManyToManyField.
- O primeiro argumento neste campo é a classe de objetos à qual este está relacionado.
- Fornecemos o argumento `blank=True`, o que significa que um passageiro pode não ter voos.
- Adicionamos um `related_name` que serve ao mesmo propósito que fez anteriormente: permitir-nos encontrar todos os passageiros em um determinado voo.

Para efetuar essas alterações, devemos fazer migrações e migrar. Podemos então registrar o modelo Passenger em `admin.py` e visitar a página do admin para criar alguns passageiros!

Agora que adicionamos alguns passageiros, vamos atualizar nossa página de voo para exibir todos os passageiros em um voo. Agora, adicionaremos uma função de reserva em `views.py` que adiciona um passageiro a um voo:

```python
def book(request, flight_id):

    # Para uma solicitação POST, adicionar um novo voo
    if request.method == "POST":

        # Acessando o voo
        flight = Flight.objects.get(pk=flight_id)

        # Encontrar o ID do passageiro nos dados do formulário enviados
        passenger_id = int(request.POST["passenger"])

        # Encontrar o passageiro com base no ID
        passenger = Passenger.objects.get(pk=passenger_id)

        # Adicionar passageiro ao voo
        passenger.flights.add(flight)

        # Redirecionar o usuário para a página do voo
        return HttpResponseRedirect(reverse("flight", args=(flight.id,)))
```

Em seguida, adicionaremos um contexto ao nosso modelo de voo para que a página tenha acesso a todos que não são atualmente passageiros no voo, usando a capacidade do Django de [excluir](https://docs.djangoproject.com/pt-br/4.0/topics/db/queries/#retrieving-specific-objects-with-filters) objetos específicos de uma consulta:

```python
def flight(request, flight_id):
    flight = Flight.objects.get(id=flight_id)
    passengers = flight.passengers.all()
    non_passengers = Passenger.objects.exclude(flights=flight).all()
    return render(request, "flights/flight.html", {
        "flight": flight,
        "passengers": passengers,
        "non_passengers": non_passengers
    })
```

```html
    {% for passenger in non_passengers %}
    <option value="{{ passenger.id }}">{{ passenger }}</option>
    {% endfor %}

</form>
```

    def index(request):

            return HttpResponseRedirect(reverse("login"))
        return render(request, "users/user.html")

Em seguida, podemos ir ao site de administração e adicionar alguns utilizadores. Depois disso, vamos voltar ao `views.py` e atualizar a nossa função `login_view` para lidar com um pedido `POST` com um nome de utilizador e palavra-passe:

    # Importações adicionais que vamos precisar:
    from django.contrib.auth import authenticate, login, logout

    def login_view(request):
        if request.method == "POST":
            # Acedendo ao nome de utilizador e palavra-passe dos dados do formulário
            username = request.POST["username"]
            password = request.POST["password"]

            # Verificar se o nome de utilizador e palavra-passe estão corretos, retornando o objeto Utilizador se estiverem
            user = authenticate(request, username=username, password=password)

            # Se o objeto Utilizador for retornado, fazer login e redirecionar para a página de índice:
            if user:
                login(request, user)
                return HttpResponseRedirect(reverse("index"))
            # Caso contrário, retornar novamente a página de login com novo contexto
            else:
                return render(request, "users/login.html", {
                    "message": "Credenciais Inválidas"
                })
        return render(request, "users/login.html")
