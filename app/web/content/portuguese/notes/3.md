- Até agora, discutimos como construir páginas da web simples usando HTML e CSS, e como usar Git e GitHub para acompanhar as mudanças em nosso código e colaborar com outras pessoas. Também nos familiarizamos com a linguagem de programação Python.

## Aplicações Web <a id="web-applications"></a>

Até agora, todas as aplicações web que escrevemos têm sido **estáticas**. Isso significa que toda vez que abrimos aquela página web, ela parece exatamente a mesma. Muitos sites que visitamos todos os dias, no entanto, mudam toda vez que os visitamos. Se você visitar os sites do [New York Times](https://www.nytimes.com/) ou [Facebook](https://www.facebook.com/), por exemplo, é muito provável que você veja coisas diferentes hoje do que verá amanhã. Para sites grandes como esses, seria irracional que os empregados tivessem que editar manualmente um grande arquivo HTML toda vez que uma mudança é feita, e é aí que os sites **dinâmicos** podem ser extremamente úteis. Um site dinâmico é aquele que aproveita uma linguagem de programação (como Python) para gerar dinamicamente arquivos HTML e CSS. Durante esta aula, vamos aprender como criar nossas primeiras aplicações dinâmicas.

## HTTP <a id="http"></a>

HTTP, ou Protocolo de Transferência de Hipertexto, é um protocolo amplamente aceito para a forma como as mensagens são transferidas para trás e para frente pela internet. Normalmente, as informações online são passadas entre um cliente (usuário) e um servidor. ![Cliente e Servidor](https://cs50.harvard.edu/web/2020/notes/3/images/client_server.png)

Neste protocolo, o cliente enviará uma **requisição** para o servidor, que pode se parecer com a seguinte. No exemplo abaixo, `GET` é simplesmente um tipo de requisição, um dos três que discutiremos neste curso. O `/` normalmente indica que estamos procurando a página inicial do site, e os três pontos indicam que poderíamos estar passando mais informações também. ![Requisição](https://cs50.harvard.edu/web/2020/notes/3/images/request.png)

Após receber uma requisição, um servidor então enviará de volta uma resposta HTTP, que pode se parecer com a seguinte. Tal resposta incluirá a versão do HTTP, um código de status (200 significa OK), uma descrição do conteúdo e algumas informações adicionais. ![Resposta](https://cs50.harvard.edu/web/2020/notes/3/images/response.png)

200 é apenas um dos muitos códigos de status, alguns dos quais você pode ter visto no passado: ![Códigos](https://cs50.harvard.edu/web/2020/notes/3/images/codes.png)

## Django <a id="django"></a>

[Django](https://www.djangoproject.com/) é um framework web baseado em Python que nos permitirá escrever código Python que gera dinamicamente HTML e CSS. A vantagem de usar um framework como o Django é que muitos códigos já estão escritos para nós e podemos tirar proveito disso.

- Para começar, precisaremos instalar o Django, o que significa que também terá que [instalar o pip](https://pip.pypa.io/en/stable/installing/) se ainda não o fez.
- Depois de instalar o Pip, você pode executar `pip3 install Django` em seu terminal para instalar o Django.

Após a instalação do Django, podemos seguir os passos para criar um novo projeto Django:

1. Execute `django-admin startproject NOME_DO_PROJETO` para criar uma série de arquivos iniciais para nosso projeto.
2. Execute `cd NOME_DO_PROJETO` para navegar até o diretório do seu novo projeto.
3. Abra o diretório no seu editor de texto de escolha. Você notará que alguns arquivos foram criados para você. Não precisaremos olhar para a maioria deles por enquanto, mas três deles serão muito importantes desde o início:
   - `manage.py` é o que usamos para executar comandos no terminal. Não precisaremos editá-lo, mas o utilizaremos com frequência.
   - `settings.py` contém algumas configurações importantes para o nosso novo projeto. Existem algumas configurações padrão, mas ocasionalmente podemos querer alterar algumas delas.
   - `urls.py` contém as direções para onde os usuários devem ser direcionados após navegarem para determinado URL.
4. Inicie o projeto executando `python manage.py runserver`. Isso abrirá um servidor de desenvolvimento, no qual você pode acessar visitando a URL fornecida. Esse servidor de desenvolvimento está sendo executado localmente em sua máquina, o que significa que outras pessoas não podem acessar seu site. Isso deve levá-lo para a página de destino padrão: ![Página de Destino](https://cs50.harvard.edu/web/2020/notes/3/images/landing.png)
5. Em seguida, teremos que criar uma aplicação. Os projetos Django são divididos em uma ou mais **aplicações**. A maioria de nossos projetos exigirá apenas uma aplicação, mas sites maiores podem se beneficiar da capacidade de dividir um site em várias apps. Para criar uma aplicação, executamos `python manage.py startapp NOME_DA_APP`. Isso criará alguns diretórios e arquivos adicionais que serão úteis em breve, incluindo `views.py`.
6. Agora, devemos instalar nossa nova aplicação. Para fazer isso, vá para `settings.py`, role para baixo até a lista de `INSTALLED_APPS` e adicione o nome de nossa nova aplicação a esta lista. ![aplicações instaladas](https://cs50.harvard.edu/web/2020/notes/3/images/installed.png)

## Rotas <a id="rotas"></a>

Agora, para começar com nossa aplicação:

1.  Em seguida, vamos navegar até `views.py`. Este arquivo conterá várias visualizações diferentes e podemos pensar em uma visualização por enquanto como uma página que o usuário possa querer ver. Para criar nossa primeira visualização, vamos escrever uma função que recebe uma `requisição`. Por enquanto, vamos simplesmente retornar um `HttpResponse` (uma resposta muito simples que inclui um código de resposta 200 e uma string de texto que pode ser exibida em um navegador da web) com "Olá, Mundo". Para fazer isso, precisamos incluir `from django.http import HttpResponse`. Nosso arquivo agora se parece com isso:

        from django.shortcuts import render
        from django.http import HttpResponse

        # Crie suas visualizações aqui.

        def index(requisição):
            return HttpResponse("Olá, mundo!")

2.  Agora, precisamos associar de alguma forma essa visualização que acabamos de criar com uma URL específica. Para fazer isso, vamos criar outro arquivo chamado `urls.py` no mesmo diretório que `views.py`. Já temos um arquivo `urls.py` para todo o projeto, mas é melhor ter um separado para cada aplicativo individual.
3.  Dentro do nosso novo `urls.py`, vamos criar uma lista de padrões de URL que um usuário pode visitar ao usar nosso site. Para fazer isso:

    1.  Temos que fazer algumas importações: `from django.urls import path` nos dará a capacidade de redirecionar URLs e `from . import views` importará quaisquer funções que tenhamos criado em `views.py`.
    2.  Crie uma lista chamada `urlpatterns`.
    3.  Para cada URL desejada, adicione um item à lista `urlpatterns` que contenha uma chamada para a função `path` com dois ou três argumentos: uma string representando o caminho da URL, uma função de `views.py` que desejamos chamar quando essa URL for visitada e (opcionalmente) um nome para esse caminho, no formato `name="alguma_coisa"`. Por exemplo, é assim que nosso aplicativo simples se parece agora:

            from django.urls import path
            from . import views

            urlpatterns = [
                path("", views.index, name="index")
            ]

4.  Agora, criamos um `urls.py` para esta aplicação específica e é hora de editar o `urls.py` criado para nós para o projeto inteiro. Quando você abrir este arquivo, você deverá ver que já existe um caminho chamado `admin` que discutiremos em palestras posteriores. Queremos adicionar outro caminho para nosso novo aplicativo, então adicionaremos um item à lista `urlpatterns`. Isso segue o mesmo padrão de nossos caminhos anteriores, exceto que, em vez de adicionar uma função de `views.py` como nosso segundo argumento, queremos ser capazes de incluir _todos_ os caminhos do arquivo `urls.py` dentro de nossa aplicação. Para fazer isso, escrevemos: `include("NOME_DO_APP.urls")`, onde `include` é uma função à qual ganhamos acesso também importando `include` de `django.urls` conforme mostrado no `urls.py` abaixo:

        from django.contrib import admin
        from django.urls import path, include

        urlpatterns = [
            path('admin/', admin.site.urls),
            path('hello/', include("hello.urls"))
        ]

5.  Ao fazer isso, especificamos que quando um usuário visita nosso site e depois na barra de endereços adiciona `/hello` à URL, eles serão redirecionados para os caminhos dentro de nossa nova aplicação.

Agora, quando inicio minha aplicação usando `python manage.py runserver` e visito a URL fornecida, me deparo com esta tela: ![url incorreto](https://cs50.harvard.edu/web/2020/notes/3/images/404.png) Mas isso ocorre porque só definimos a URL `localhost:8000/hello`, mas não definimos a URL `localhost:8000` sem nada adicionado no final. Portanto, quando adiciono `/hello` à URL na barra de endereços: ![Olá, mundo](https://cs50.harvard.edu/web/2020/notes/3/images/helloworld.png) Agora que tivemos algum sucesso, vamos revisar o que aconteceu para chegarmos a esse ponto:

1. Quando acessamos a URL `localhost:8000/hello/`, o Django verificou o que veio após a URL base (`localhost:8000/`) e foi ao arquivo `urls.py` do nosso projeto e procurou por um padrão que correspondesse a `hello`.
2. Encontrou essa extensão porque a definimos e viu que, ao encontrar essa extensão, deveria `incluia` nosso arquivo `urls.py` de dentro de nossa aplicação.
3. Em seguida, o Django ignorou as partes da URL que já foram usadas na reconfiguração (`localhost:8000/hello/`, ou tudo) e procurou dentro de nosso outro arquivo `urls.py` por um padrão que correspondesse à parte restante da URL.
4. Encontrou que nosso único caminho até agora (`""`) correspondia ao que restava da URL, e assim nos direcionou para a função de `views.py` associada a esse caminho.
5. Finalmente, o Django executou aquela função dentro de `views.py` e retornou o resultado (`HttpResponse("Olá, mundo!")`) para nosso navegador da web.

Agora, se quisermos, podemos alterar a função `index` dentro de `views.py` para retornar o que quisermos! Poderíamos até manter o controle de variáveis e fazer cálculos dentro da função antes de eventualmente retornar algo.

Agora, vamos ver como podemos adicionar mais de uma visualização à nossa aplicação. Podemos seguir muitas das mesmas etapas dentro de nossa aplicação para criar páginas que digam olá para Brian e David.

Dentro de `views.py`:

    from django.shortcuts import render
    from django.http import HttpResponse

    # Crie suas visualizações aqui.

    def index(requisição):
        return HttpResponse("Olá, mundo!")

    def brian(requisição):
        return HttpResponse("Olá, Brian!")

    def david(requisição):
        return HttpResponse("Olá, David!")

Dentro de `urls.py` (dentro de nossa aplicação):

    from django.urls import path
    from . import views

    urlpatterns = [
        path("", views.index, name="index"),
        path("brian", views.brian, name="brian"),
        path("david", views.david, name="david")
    ]

Agora, nosso site permanece inalterado quando visitamos `localhost:8000/hello`, mas obtemos páginas diferentes quando adicionamos `brian` ou `david` à URL: ![Brian](https://cs50.harvard.edu/web/2020/notes/3/images/brian.png) ![David](https://cs50.harvard.edu/web/2020/notes/3/images/david.png)

Muitos sites são parametrizados por itens incluídos na URL. Por exemplo, acessar [www.twitter.com/cs50](https://twitter.com/cs50) mostrará todos os tweets do CS50 e acessar [www.github.com/cs50](https://github.com/cs50) o levará à página do GitHub do CS50. Você até pode encontrar seus próprios repositórios públicos do GitHub navegando para `www.github.com/SEU_NOME_DE_USUÁRIO`!

Ao pensar sobre como isso é implementado, parece impossível que sites como GitHub e Twitter tenham um caminho de URL individual para cada um de seus usuários, então vamos ver como poderíamos fazer um caminho mais flexível. Vamos começar adicionando uma função mais geral, chamada `greet`, em `views.py`:

    def greet(requisição, nome):
        return HttpResponse(f"Olá, {nome}!")

Esta função recebe não apenas uma solicitação, mas também um argumento adicional com o nome do usuário e depois retorna uma resposta HTTP personalizada com base nesse nome. Em seguida, temos que criar um caminho mais flexível em `urls.py`, que poderia se parecer com isso:

    path("<str:nome>", views.greet, name="greet")

Esta é uma nova sintaxe, mas essencialmente o que está acontecendo aqui é que não estamos mais procurando por uma palavra ou nome específico na URL, mas por qualquer string que um usuário possa inserir. Agora, podemos experimentar o site com algumas outras URLs: ![harry](https://cs50.harvard.edu/web/2020/notes/3/images/harry.png) ![connor](https://cs50.harvard.edu/web/2020/notes/3/images/connor.png)

Eu até posso tornar isso um pouco mais agradável, aumentando a função `greet` para utilizar a função `capitalize` do Python que capitaliza uma string:

    def greet(requisição, nome):
        return HttpResponse(f"Olá, {nome.capitalize()}!")

![Harry](https://cs50.harvard.edu/web/2020/notes/3/images/harryc.png) ![Connor](https://cs50.harvard.edu/web/2020/notes/3/images/connorc.png)

Essa é uma ótima ilustração de como qualquer funcionalidade que temos em Python pode ser usada no Django antes de ser retornada.

## Modelos <a id="templates"></a>

Até agora, nossas Respostas HTTP foram apenas texto, mas podemos incluir quaisquer elementos HTML que desejarmos! Por exemplo, eu poderia decidir retornar um cabeçalho azul em vez de apenas o texto em nossa função `index`:

    def index(request):
        return HttpResponse("<h1 style=\"color:blue\">Olá, mundo!</h1>")

![Azul](https://cs50.harvard.edu/web/2020/notes/3/images/bluehello.png)

Seria muito tedioso escrever uma página HTML inteira dentro do `views.py`. Também seria considerado um design ruim, pois queremos manter partes separadas do nosso projeto em arquivos separados sempre que possível.

É por isso que agora vamos introduzir [os modelos do Django](https://docs.djangoproject.com/en/4.0/topics/templates/), que nos permitirão escrever HTML e CSS em arquivos separados e renderizar esses arquivos usando o Django. A sintaxe que usaremos para renderizar um modelo é a seguinte:

    def index(request):
        return render(request, "hello/index.html")

Agora, precisaremos criar esse modelo. Para fazer isso, criaremos uma pasta chamada `templates` dentro do nosso aplicativo, em seguida criaremos uma pasta chamada `hello` (ou qualquer que seja o nome do nosso aplicativo) dentro dela, e então adicionaremos um arquivo chamado `index.html`.

![Arquivos](https://cs50.harvard.edu/web/2020/notes/3/images/files.png)

Em seguida, adicionaremos o que desejamos a esse novo arquivo:

    <!DOCTYPE html>
    <html lang="en">
        <head>
            <title>Olá</title>
        </head>
        <body>
            <h1>Olá, Mundo!</h1>
        </body>
    </html>

Agora, quando visitamos a página principal de nosso aplicativo, podemos ver que o cabeçalho e o título foram atualizados: ![template0](https://cs50.harvard.edu/web/2020/notes/3/images/template0.png)

Além de escrever algumas páginas HTML estáticas, também podemos usar [a linguagem de modelagem do Django](https://docs.djangoproject.com/en/4.0/ref/templates/language/) para alterar o conteúdo de nossos arquivos HTML com base na URL visitada. Vamos tentar mudar nossa função `greet` anterior:

```python
def greet(request, name):
    return render(request, "hello/greet.html", {
        "name": name.capitalize()
    })
```

Observe que passamos um terceiro argumento para a função `render` aqui, conhecido como **contexto**. Neste contexto, podemos fornecer informações que gostaríamos de ter disponíveis em nossos arquivos HTML. Esse contexto tem a forma de um dicionário Python. Agora, podemos criar um arquivo `greet.html`:

    <!DOCTYPE html>
    <html lang="en">
        <head>
            <title>Olá</title>
        </head>
        <body>
            <h1>Olá, {{ name }}!</h1>
        </body>
    </html>

Você notará que usamos uma nova sintaxe: duplas chaves. Essa sintaxe nos permite acessar variáveis que fornecemos no argumento `context`. Agora, quando testamos: ![Template 1](https://cs50.harvard.edu/web/2020/notes/3/images/template1.png) ![Template 2](https://cs50.harvard.edu/web/2020/notes/3/images/template2.png)

Agora, vimos como podemos modificar nossos modelos HTML com base no contexto que fornecemos. No entanto, a linguagem de modelagem do Django é ainda mais poderosa do que isso, então vamos dar uma olhada em algumas outras maneiras em que ela pode ser útil:

### Condicionais <a id="conditionals"></a>

Podemos querer alterar o que é exibido em nosso site dependendo de algumas condições. Por exemplo, se visitarmos o site [www.isitchristmas.com](https://www.isitchristmas.com), provavelmente nos depararemos com uma página assim: ![no](https://cs50.harvard.edu/web/2020/notes/3/images/isitno.png) Mas este site mudará no dia de Natal, quando ele dirá **SIM**. Para criar algo semelhante para nós, vamos tentar criar um aplicativo similar, onde verificamos se é o Dia de Ano Novo. Vamos criar um novo aplicativo para fazer isso, seguindo o processo de criar um novo aplicativo:

1.  execute `python manage.py startapp newyear` no terminal.
2.  Edite `settings.py`, adicionando "newyear" como um dos `INSTALLED_APPS`
3.  Edite o arquivo `urls.py` do nosso projeto e inclua um caminho semelhante ao que criamos para o aplicativo `hello`:

```
path('newyear/', include("newyear.urls"))
```

4.  Crie outro arquivo `urls.py` dentro do diretório do nosso novo aplicativo e atualize-o para incluir um caminho semelhante ao caminho de índice em `hello`:

```
from django.urls import path
from . import views

urlpatterns = [
    path("", views.index, name="index"),
]
```

5.  Crie uma função de índice em `views.py`.

Agora que estamos configurados com nosso novo aplicativo, vamos descobrir como verificar se é o Dia de Ano Novo ou não. Para fazer isso, podemos importar o módulo [datetime](https://docs.python.org/3/library/datetime.html) do Python. Para ter uma ideia de como esse módulo funciona, podemos olhar a [documentação](https://docs.python.org/3/library/datetime.html) e então testá-lo fora do Django usando o interpretador Python.

- O **interpretador Python** é uma ferramenta que podemos usar para testar pequenos trechos de código Python. Para usar isso, execute `python` no seu terminal e então poderá digitar e executar código Python em seu terminal. Quando terminar de usar o interpretador, execute `exit()` para sair. ![interpretador](https://cs50.harvard.edu/web/2020/notes/3/images/datetime.png)
- Podemos usar esse conhecimento para construir uma expressão booleana que avaliará como Verdadeira somente se hoje for o Dia de Ano Novo: `now.day == 1 and now.month == 1`
- Agora que temos uma expressão que podemos usar para verificar se é o Dia de Ano Novo ou não, podemos atualizar nossa função de índice em `views.py`:

```python
def index(request):
    now = datetime.datetime.now()
    return render(request, "newyear/index.html", {
        "newyear": now.month == 1 and now.day == 1
    })
```

Agora, vamos criar nosso modelo `index.html`. Teremos que novamente criar uma nova pasta chamada `templates`, uma pasta dentro dela chamada `newyear` e um arquivo chamado `index.html` dentro dessa pasta. Dentro desse arquivo, vamos escrever algo assim:

    <!DOCTYPE html>
    <html lang="en">
        <head>
            <title>É Ano Novo?</title>
        </head>
        <body>
            {% if newyear %}
                <h1>SIM</h1>
            {% else %}
                <h1>NÃO</h1>
            {% endif %}
        </body>
    </html>

No código acima, observe que quando desejamos incluir lógica em nossos arquivos HTML, usamos `{%` e `%}` como tags de abertura e fechamento ao redor das declarações lógicas. Observe também que a linguagem de formatação do Django requer que você inclua uma tag de finalização indicando que terminamos nosso bloco `if-else`. Agora, podemos abrir nossa página para ver:

![Não](https://cs50.harvard.edu/web/2020/notes/3/images/no.png)

Agora, para termos uma ideia melhor do que está acontecendo nos bastidores, vamos inspecionar o elemento desta página:

![Fonte](https://cs50.harvard.edu/web/2020/notes/3/images/source.png)

Observe que o HTML que está sendo enviado para o seu navegador inclui apenas o cabeçalho de NÃO, o que significa que o Django está usando o modelo HTML que escrevemos para criar um novo arquivo HTML e então enviá-lo para o nosso navegador. Se trapacearmos um pouco e garantirmos que nossa condição seja sempre verdadeira, veremos que o caso oposto é preenchido:

    def index(request):
        now = datetime.datetime.now()
        return render(request, "newyear/index.html", {
            "newyear": True
        })

![Sim](https://cs50.harvard.edu/web/2020/notes/3/images/yes.png) ![Fonte 0](https://cs50.harvard.edu/web/2020/notes/3/images/source0.png)

### Estilização <a id="styling"></a>

Se quisermos adicionar um arquivo CSS, que é um arquivo _estático_ porque não muda, primeiro criaremos uma pasta chamada `static`, em seguida criamos uma pasta `newyear` dentro dela e então um arquivo `styles.css` dentro disso. Neste arquivo, podemos adicionar qualquer estilo que desejarmos, assim como fizemos na primeira aula:

    h1 {
        font-family: sans-serif;
        font-size: 90px;
        text-align: center;
    }

Agora, para incluir esse estilo em nosso arquivo HTML, adicionamos a linha `{% load static %}` no topo do nosso modelo HTML, sinalizando ao Django que desejamos ter acesso aos arquivos em nossa pasta `static`. Em seguida, ao invés de codificar o link para uma folha de estilo como fizemos anteriormente, usaremos uma sintaxe específica do Django:

    <link rel="stylesheet" href="{% static 'newyear/styles.css' %}">

Agora, se reiniciarmos o servidor, veremos que as alterações de estilo foram aplicadas: ![grande não](https://cs50.harvard.edu/web/2020/notes/3/images/bigno.png)

## Tarefas <a id="tasks"></a>

Agora, vamos aplicar o que aprendemos até agora em um mini projeto: criando uma lista de tarefas (TODO list). Vamos começar, mais uma vez, criando um novo aplicativo:

1.  Execute `python manage.py startapp tasks` no terminal.
2.  Edite `settings.py`, adicionando “tasks” como um dos nossos `INSTALLED_APPS`.
3.  Edite o arquivo `urls.py` do nosso projeto e inclua um caminho semelhante ao que criamos para o aplicativo `hello`:

         path('tasks/', include("tasks.urls"))

4.  Crie outro arquivo `urls.py` dentro do diretório do nosso novo aplicativo e atualize-o para incluir um caminho semelhante ao caminho do índice em `hello`:

         from django.urls import path
         from . import views

         urlpatterns = [
             path("", views.index, name="index"),
         ]

5.  Crie uma função de índice em `views.py`.

Agora, vamos começar tentando simplesmente criar uma lista de tarefas e em seguida exibi-las em uma página. Vamos criar uma lista em Python no início de `views.py` onde armazenaremos nossas tarefas. Em seguida, podemos atualizar nossa função `index` para renderizar um modelo e fornecer nossa lista recém-criada como contexto.

    from django.shortcuts import render

    tasks = ["foo", "bar", "baz"]

    # Crie suas views aqui.
    def index(request):
        return render(request, "tasks/index.html", {
            "tasks": tasks
        })

Agora, vamos trabalhar na criação do nosso arquivo de modelo HTML:

    <!DOCTYPE html>
    <html lang="en">
        <head>
            <title>Tarefas</title>
        </head>
        <body>
            <ul>
                {% for task in tasks %}
                    <li>{{ task }}</li>
                {% endfor %}
            </ul>
        </body>
    </html>

Observe que podemos fazer um loop sobre nossas tarefas usando uma sintaxe semelhante às nossas condicionais anteriores e também semelhante a um loop Python da Aula 2. Quando vamos para a página de tarefas agora, podemos ver nossa lista sendo renderizada: ![tasks0](https://cs50.harvard.edu/web/2020/notes/3/images/tasks0.png)

## Formulários <a id="forms"></a>

Agora que podemos visualizar todas as nossas tarefas atuais como uma lista, talvez queiramos ser capazes de adicionar novas tarefas. Para fazer isso, vamos começar a usar formulários para atualizar uma página da web. Vamos começar adicionando uma nova função ao `views.py` que renderizará uma página com um formulário para adicionar uma nova tarefa:

    # Adicionar uma nova tarefa:
    def add(request):
        return render(request, "tasks/add.html")

Em seguida, certifique-se de adicionar um novo caminho ao `urls.py`:

    path("add", views.add, name="add")

Agora, vamos criar nosso arquivo `add.html`, que é bastante semelhante ao `index.html`, exceto que no corpo incluiremos um formulário em vez de uma lista:

    <!DOCTYPE html>
    <html lang="en">
        <head>
            <title>Tarefas</title>
        </head>
        <body>
            <h1>Adicionar Tarefa:</h1>
            <form action="">
                <input type="text" name="task">
                <input type="submit">
            </form>
        </body>
    </html>

Entretanto, o que acabamos de fazer não é necessariamente o melhor design, pois repetimos a maior parte desse HTML em dois arquivos diferentes. A linguagem de modelagem do Django nos dá uma forma de eliminar esse design deficiente: [herança de template](https://tutorial.djangogirls.org/pt/template_extending/). Isso nos permite criar um arquivo `layout.html` que conterá a estrutura geral de nossa página:

    <!DOCTYPE html>
    <html lang="en">
        <head>
            <title>Tarefas</title>
        </head>
        <body>
            {% block body %}
            {% endblock %}
        </body>
    </html>

Observe que novamente usamos `{%...%}` para denotar algum tipo de lógica não-HTML e, neste caso, estamos dizendo ao Django para preencher esse "bloco" com algum texto de outro arquivo. Agora, podemos alterar nossos outros dois arquivos HTML para ficarem assim:

`index.html`:

```xml
{% extends "tasks/layout.html" %}

{% block body %}
    <h1>Tarefas:</h1>
    <ul>
        {% for task in tasks %}
            <li>{{ task }}</li>
        {% endfor %}
    </ul>
{% endblock %}
```

`add.html`:

```xml
{% extends "tasks/layout.html" %}

{% block body %}
    <h1>Adicionar Tarefa:</h1>
    <form action="">
        <input type="text" name="task">
        <input type="submit">
    </form>
{% endblock %}
```

Observe como agora podemos eliminar grande parte do código repetido ao _estender_ nosso arquivo de layout. Agora, nossa página de índice permanece a mesma, e agora temos uma página de adição também:

![Adicionar](https://cs50.harvard.edu/web/2020/notes/3/images/add.png)

Em seguida, não é ideal ter que digitar "/add" na URL toda vez que quisermos adicionar uma nova tarefa, então provavelmente queremos adicionar alguns links entre as páginas. Em vez de codificar os links, podemos agora usar a variável `name` que atribuímos a cada caminho em `urls.py` e criar um link que se parece com este:

```xml
<a href="{% url 'add' %}">Adicionar uma Nova Tarefa</a>
```

onde 'add' é o nome desse caminho. Podemos fazer algo semelhante no nosso `add.html`:

```xml
<a href="{% url 'index' %}">Visualizar Tarefas</a>
```

Isso poderia potencialmente criar um problema, já que temos algumas rotas nomeadas `index` em nossos diferentes aplicativos. Podemos resolver isso acessando o arquivo `urls.py` de cada um de nossos aplicativos e adicionando uma variável `app_name`, de modo que os arquivos agora se pareçam com isso:

    from django.urls import path
    from . import views

    app_name = "tasks"
    urlpatterns = [
        path("", views.index, name="index"),
        path("add", views.add, name="add")
    ]

Em seguida, alteramos nossos links de simplesmente `index` e `add` para `tasks:index` e `tasks:add`

    <a href="{% url 'tasks:index' %}">Visualizar Tarefas</a>

    <a href="{% url 'tasks:add' %}">Adicionar uma Nova Tarefa</a>

Agora, vamos trabalhar para garantir que o formulário realmente faça algo quando o usuário o envia. Podemos fazer isso adicionando uma `action` ao formulário que criamos em `add.html`:

```xml
<form action="{% url 'tasks:add' %}" method="post">
```

Isso significa que uma vez que o formulário for enviado, seremos redirecionados de volta para a URL `add`. Aqui especificamos que estaremos usando um método _post_ em vez de um método _get_, que é tipicamente o que usamos sempre que um formulário pode alterar o estado da página da web.

Agora precisamos adicionar um pouco mais a este formulário, porque o Django requer um token para evitar [Ataque de Falsificação de Solicitação entre Sites (CSRF)](https://portswigger.net/web-security/csrf). Este é um ataque onde um usuário malicioso tenta enviar uma solicitação para seu servidor de algum lugar que não seja seu site. Isso poderia ser um problema muito grande para alguns sites. Por exemplo, digamos que um site bancário tenha um formulário para um usuário transferir dinheiro para outro. Seria catastrófico se alguém pudesse enviar uma transferência de fora do site do banco!

Para resolver esse problema, quando o Django envia uma resposta renderizando um modelo, ele também fornece um **token CSRF** único em cada nova sessão no site. Em seguida, quando uma solicitação é enviada, o Django verifica se o token CSRF associado à solicitação corresponde a um que ele forneceu recentemente. Portanto, se um usuário malicioso em outro site tentasse enviar uma solicitação, ele seria bloqueado devido a um token CSRF inválido. Essa validação CSRF está integrada ao [Framework de Middleware do Django](https://docs.djangoproject.com/pt/4.0/topics/http/middleware/), que pode intervir no processamento de solicitação-resposta de um aplicativo Django. Não entraremos em mais detalhes sobre o Middleware neste curso, mas dê uma olhada na [documentação](https://docs.djangoproject.com/pt/4.0/topics/http/middleware/) se estiver interessado!

Para incorporar essa tecnologia em nosso código, precisamos adicionar uma linha ao nosso formulário em `add.html`.

    <form action="{% url 'tasks:add' %}" method="post">
        {% csrf_token %}
        <input type="text" name="task">
        <input type="submit">
    </form>

Essa linha adiciona um campo de entrada oculto com o token CSRF fornecido pelo Django, de modo que quando recarregamos a página, parece que nada mudou. No entanto, se inspecionarmos o elemento, perceberemos que um novo campo de entrada foi adicionado: ![CSRF](https://cs50.harvard.edu/web/2020/notes/3/images/csrf.png)

### Formulários do Django <a id="django-forms"></a>

Enquanto podemos criar formulários escrevendo HTML puro como acabamos de fazer, o Django fornece uma maneira ainda mais fácil de coletar informações de um usuário: [Formulários do Django](https://docs.djangoproject.com/pt/4.0/ref/forms/api/). Para usar este método, vamos adicionar o seguinte no topo do `views.py` para importar o módulo `forms`:

    from django import forms

Agora, podemos criar um novo formulário dentro do `views.py` criando uma classe Python chamada `NewTaskForm`:

    class NewTaskForm(forms.Form):
        task = forms.CharField(label="Nova Tarefa")

Agora, vamos analisar o que está acontecendo nessa classe:

- Dentro dos parênteses após `NewTaskForm`, vemos que temos `forms.Form`. Isso ocorre porque nosso novo formulário [herda](https://www.w3schools.com/python/python_inheritance.asp) de uma classe chamada `Form` que está incluída no módulo `forms`. Já vimos como a herança pode ser usada na linguagem de modelagem do Django e para estilização usando Sass. Este é outro exemplo de como a herança é usada para pegar uma descrição mais geral (a classe `forms.Form`) e estreitá-la para o que queremos (nosso novo Formulário). A herança é uma parte importante da Programação Orientada a Objetos que não discutiremos em detalhes durante este curso, mas existem [muitos recursos online](https://www.w3schools.com/python/python_inheritance.asp) disponíveis para aprender sobre o assunto!
- Dentro desta classe, podemos especificar que tipo de informação gostaríamos de coletar do usuário, neste caso o nome de uma tarefa.
- Especificamos que isso deve ser uma entrada textual escrevendo `forms.CharField`, mas há [muitos outros campos de entrada](https://docs.djangoproject.com/pt/4.0/ref/forms/fields/#built-in-field-classes) incluídos no módulo de formulário do Django que podemos escolher.
- Dentro deste `CharField`, especificamos um `label`, que aparecerá para o usuário quando eles carregarem a página. Um `label` é apenas um dos [muitos argumentos](https://docs.djangoproject.com/pt/4.0/ref/forms/fields/#core-field-arguments) que podemos passar para um campo de formulário.

Agora que criamos a classe `NewTaskForm`, podemos incluí-la no contexto ao renderizar a página `add`:

```python
# Adicionar uma nova tarefa:
def add(request):
    return render(request, "tasks/add.html", {
        "form": NewTaskForm()
    })
```

Agora, em `add.html`, podemos substituir nosso campo de entrada pelo formulário que acabamos de criar:

```xml
{% extends "tasks/layout.html" %}

{% block body %}
    <h1>Adicionar Tarefa:</h1>
    <form action="{% url 'tasks:add' %}" method="post">
        {% csrf_token %}
        {{ form }}
        <input type="submit">
    </form>
    <a href="{% url 'tasks:index' %}">Visualizar Tarefas</a>
{% endblock %}
```

Existem várias vantagens em usar o módulo `forms` em vez de escrever manualmente um formulário HTML:

- Se quisermos adicionar novos campos ao formulário, podemos simplesmente adicioná-los em `views.py` sem digitar HTML adicional.
- O Django realiza automaticamente a [validação do lado do cliente](https://developer.mozilla.org/pt-BR/docs/Learn/Forms/Form_validation) ou validação local na máquina do usuário, o que significa que não permitirá que um usuário envie seu formulário se estiver incompleto.
- O Django fornece uma simples [validação no lado do servidor](https://developer.mozilla.org/pt-BR/docs/Learn/Forms/Form_validation) ou validação que ocorre assim que os dados do formulário atingem o servidor.
- Na próxima aula, começaremos a usar os **modelos** para armazenar informações, e o Django torna muito simples criar um formulário com base em um modelo.

Agora que temos um formulário configurado, vamos trabalhar no que acontece quando um usuário clica no botão de envio. Quando um usuário navega até a página de adição clicando em um link ou digitando a URL, ele envia uma solicitação `GET` para o servidor, que já tratamos em nossa função `add`. No entanto, quando um usuário envia um formulário, ele envia uma solicitação `POST` para o servidor, que no momento não é tratada na função `add`. Podemos lidar com um método `POST` adicionando uma condição com base no argumento `request` que nossa função recebe. Os comentários no código abaixo explicam o propósito de cada linha:

```python
# Adicionar uma nova tarefa:
def add(request):

    # Verificar se o método é POST
    if request.method == "POST":

        # Pegar os dados que o usuário enviou e salvá-los como formulário
        form = NewTaskForm(request.POST)

        # Verificar se os dados do formulário são válidos (lado servidor)
        if form.is_valid():

            # Isolar a tarefa da versão 'limpa' dos dados do formulário
            task = form.cleaned_data["task"]

            # Adicionar a nova tarefa à nossa lista de tarefas
            tasks.append(task)

            # Redirecionar o usuário para a lista de tarefas
            return HttpResponseRedirect(reverse("tasks:index"))

        else:

            # Se o formulário for inválido, renderizar novamente a página com as informações existentes.
            return render(request, "tasks/add.html", {
                "form": form
            })

    return render(request, "tasks/add.html", {
        "form": NewTaskForm()
    })
```

Uma observação rápida: para redirecionar o usuário após um envio bem-sucedido, precisamos de algumas importações adicionais:

    from django.urls import reverse
    from django.http import HttpResponseRedirect

## Sessões <a id="sessions"></a>

Neste ponto, nós construímos com sucesso uma aplicação que nos permite adicionar tarefas a uma lista crescente. No entanto, pode ser um problema armazenar essas tarefas como uma variável global, pois isso significa que todos os usuários que visitam a página veem exatamente a mesma lista. Para resolver este problema, vamos utilizar uma ferramenta conhecida como [sessões.](https://docs.djangoproject.com/en/4.0/topics/http/sessions/)

Sessões são uma maneira de armazenar dados únicos no lado do servidor para cada nova visita a um site.

Para usar sessões em nossa aplicação, primeiro vamos excluir nossa variável global `tasks`, depois alterar nossa função `index`, e por fim garantir que em qualquer outro lugar em que tivéssemos usado a variável `tasks`, a substituímos por `request.session["tasks"]`.

```python
def index(request):

    # Verificar se já existe a chave "tasks" em nossa sessão

    if "tasks" not in request.session:

        # Se não houver, criar uma nova lista
        request.session["tasks"] = []

    return render(request, "tasks/index.html", {
        "tasks": request.session["tasks"]
    })

# Adicionar uma nova tarefa:
def add(request):
    if request.method == "POST":

        # Pegar os dados enviados pelo usuário e salvá-los como form
        form = NewTaskForm(request.POST)

        # Verificar se os dados do formulário são válidos (no lado do servidor)
        if form.is_valid():

            # Isolar a tarefa a partir da versão 'limpa' dos dados do formulário
            task = form.cleaned_data["task"]

            # Adicionar a nova tarefa à nossa lista de tarefas
            request.session["tasks"] += [task]

            # Redirecionar o usuário para a lista de tarefas
            return HttpResponseRedirect(reverse("tasks:index"))
        else:

            # Se o formulário não for válido, renderizar a página novamente com as informações existentes.
            return render(request, "tasks/add.html", {
                "form": form
            })

    return render(request, "tasks/add.html", {
        "form": NewTaskForm()
    })
```

Por fim, antes que o Django consiga armazenar esses dados, devemos executar `python manage.py migrate` no terminal. Na próxima semana, falaremos mais sobre o que é uma migração, mas por agora saiba que o comando acima nos permite armazenar sessões.

Isso conclui esta aula! Na próxima vez, estaremos trabalhando em usar o Django para armazenar, acessar e manipular dados.
